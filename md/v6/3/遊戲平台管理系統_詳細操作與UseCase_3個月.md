# 遊戲平台管理系統 (MVP Phase 1) - 方案 B：標準進階版 (3個月)
## 詳細操作流程與 Use Case 說明

相較於方案 C 的區域單點營運，方案 B 主打的是「具備展店能力與第三方加盟」的標準版。增加了視覺化營運工具與外部金流的能力。

---

## Use Case 1: 代理商拓點與獨立看板 (Agent Expansion)
**【情境說明】** 老闆 (超級管理員) 決定將設備租給台中的合作方 (代理商)。需要確保台中代理商只能看到自己的數據，不會看到台北本店的營收。
**【主要角色】** 總部老闆、台中代理商

1. **老闆建立權限**：老闆登入後台，進入「使用者與權限」，新增一組帳號 `Taichung_Boss`，角色選定為 `Agent (代理商)`。
2. **分配機台綁定**：老闆將出貨到台中的 10 台機器，在系統中批量選取，點擊 [轉移歸屬] -> 指定給 `Taichung_Boss`。
3. **代理商獨立視角**：台中代理商登入後台。他看到的儀表板 (Dashboard) 上的「總營收 $50,000」與「圓餅圖」，**完全只由他名下的那 10 台機器加總而來**。
4. **子帳號開設**：台中代理商自己進入「權限管理」，建立兩組帳號給他請的早班與晚班員工。這兩組帳號的權限被自動鎖死在「只能開洗分，無法看總報表」。

## Use Case 2: 玩家跨平台資金轉移 (Third-party Wallet Transfer)
**【情境說明】** 系統開始串接外部流行的捕魚機或老虎機遊戲。玩家想用本地的錢去玩外部廠商的遊戲。
**【主要角色】** 現場玩家、V6 系統、外部遊戲商 API

1. 玩家在機台入座，此時他的中心錢包有 `$10,000`。
2. 玩家在螢幕上點擊【PG電子 - 麻將胡了】的圖標。
3. V6 系統跳出提示「正在切換遊戲空間...」，此時 V6 呼叫第三方 API `POST /deposit`，並將 `$10,000` 全數轉入 PG 平台。
4. V6 資料庫將該筆資金標記為 `[已進入 PG 電子]`，中心錢包顯示 `$0`。
5. 玩家在 PG 的遊戲體系內遊玩了 2 小時，贏了錢，總額變為 `$12,000`。
6. 玩家點擊畫面上的【離開遊戲，退回大廳】。
7. V6 收到指令，向 PG 呼叫 `POST /withdraw` 回收點數。
8. V6 確認收到 `$12,000` 後，更新中心錢包，畫面顯示 `$12,000`。
9. *(若轉帳過程發生網路連線失敗 Timeout，該筆帳目會在後台被高亮標示為🔴異常交易，提醒老闆進後台手動處理。)*

## Use Case 3: 機台斷線的即時警報 (Heartbeat Monitoring)
**【情境說明】** 台中店鋪的網路數據機被客人不小心踢掉插頭了。
**【主要角色】** 台中代理商、機台本體

1. 機台原先每 60 秒會自動發送一次含有「在線狀態、CPU熱度」的 Heartbeat 到中央。
2. 插頭被踢掉後，中央伺服器經過 3 次週期 (約 3 分鐘) 沒收到任何 Ping。
3. 台中代理商的管理看板上，原本綠色的機台燈號立刻變成了🔴紅色，並跳出系統提醒「警告：有 1 台設備失去連線」。
4. 代理商打電話給早班員工：「看一下門口左邊那台是不是沒電或沒網路了？」。員工接上插頭。
5. 網路恢復後，機台恢復發送 Ping。後台燈號自動切換回🟢綠色。
6. (同時，斷線期間的本地交易，會依照 基礎商用版的保護機制，在連線後自動排入背景佇列重新上傳，確保帳目依然正確)。

# 遊戲平台管理系統 (MVP Phase 1) - 方案 C：基礎商用版 (2個月)
## 詳細操作流程與 Use Case 說明

本版本定義為「無痛落地的基礎商用 SaaS」。告別了人工改 Code 與掉資料的窘境，本文件具象化描述店長、現場人員在日常營運中如何與系統穩定地互動。

---

## Use Case 1: 現場穩定的開洗分操作 (Secure Local Cash-in/out)
**【情境說明】** 現場人員在實體機台上，替玩家加分與洗分。必須確保即使拔掉網路線，依然能順利運作且對帳無誤。
**【主要角色】** 現場操作員、機台本地伺服器

### 階段一：正常操作
1. 操作員於機台隱藏介面輸入 4 碼 PIN 碼解鎖（取代過去毫無防護的設計）。
2. 在數字鍵盤上輸入 `1000` 並按下 `[注資]`。
3. 系統將 `$1,000` 存入 SQLite，並顯示綠色勾勾「✅ 已成功進分1,000，並已同步至中央」。

### 階段二：離線斷網保護
1. 若展場或店面網路意外斷線，操作員一樣按下 `[注資]`。
2. 系統將 `$1,000` 存入 SQLite，此時會顯示黃色時鐘「⏳ 已成功進分1,000，(網路離線中，已排入待傳佇列)」。
3. 玩家依然可以立刻使用這 $1,000 開始打遊戲，現場營運完全不中斷。
4. 網路恢復後，背景的 Worker 會自動將該筆紀錄推送到中央，且前端畫面的「待傳佇列」數字會歸零。

## Use Case 2: 本地機台連線設定 (Local Setup UI)
**【情境說明】** 將新機台搬到新店面，工程師不需要再用記事本去改 `.env`，可以優雅地透過 UI 完成初步綁定。
**【主要角色】** 裝機工程師

1. 工程師將主機接上螢幕與鍵盤，打開瀏覽器進入 `http://localhost:3000/setup`。
2. 系統要求輸入預設的工程密碼。
3. 登入後，畫面呈現兩個簡單的輸入框：
   * **中央伺服器 API**：填入 `https://api.my-game-platform.com`
   * **機台專屬金鑰 (Machine Token)**：填入從中央後台產生出的一串金鑰 `M1-XYZ-1234`。
4. 點擊「儲存並綁定」，系統自動與中央請求連線測試。
5. 測試成功後，系統自動重啟本地 Node.js 服務，機台即刻處於可營業狀態。

## Use Case 3: 中央查詢與對帳 (Central Reconciliation)
**【情境說明】** 每天晚上店長要核對報表。
**【主要角色】** 店長 (Admin)

1. 店長登入中央管理網頁。
2. 進入「平台交易總表」。
3. 透過上方的 DataGrid，選擇日期 `今日`，系統便會列出這幾百筆的清單。
4. 表格下方會自動幫店長 Sum (加總) 出「總洗分」、「總開分」，以及「盈虧數字」。
5. 如果有客人爭議說某筆錢是錯的，店長可以透過「交易時間與機台ID」快速利用欄位排序與 Filter 找到該筆帳目，雖然不允許修改金額，但店長可以點選該筆帳款加入備註：「這筆客訴，現金退還 $50」。

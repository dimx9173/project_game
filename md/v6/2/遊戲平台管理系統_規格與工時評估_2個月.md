# 遊戲平台管理系統 (MVP Phase 1) - 方案 C：基礎商用版 (2個月)
**開發時程：2 個月 (約 60 工作天/Man-Days)**  
**預算規模：40 萬**

## 一、方案定位與預期目標
本方案定位為**「快速落地、安全可靠的第一線商用版本」**。相比於傳統粗糙的 POC，本方案去除了華而不實的進階圖表與複雜權限，但**堅守資料不遺失、設定可視化**的底線。非常適合需要在 2 個月內立刻投入實體展場或店面營業，且要求系統不能像「拼裝車」的客戶。

## 二、詳細功能規格與系統升級點 (相較於破爛版)

### 2.1 核心資料與斷線保單機制 (預估工時: 12 MD)
* **資料不遺失承諾**：
  * 機台端 (SQLite) 開/洗分後，會將跨網路上傳的任務存入本地的「背景佇列 (Background Queue)」。
  * **斷線重試**：若網路斷線，前台操作完全不受阻礙。背景 Queue 會持續每 3 分鐘自動重試，直到中央伺服器 (PostgreSQL) 成功回應為止。
* **無第三方遊戲串接**：
  * 專注於本地自製遊戲引擎的對接。本階段不拉取任何外部遊戲供應商的 API，大幅降低查帳複雜度。

### 2.2 集中式後台 (Central Backend) (預估工時: 20 MD)
* **雙層實體權限 (RBAC)**
  * 提供「超級管理員 (Admin)」與「一般操作員 (Operator)」兩層登入。告別全系統共用一組密碼的窘境。
  * 操作員只能查看報表與基礎操作，無法匯出名單或更改機台全域設定。
* **正規化數據列表 (DataGrids)**
  * 不提供複雜圖表 (無 Pie Chart / Bar Chart)，但提供具備「分頁、日期區間過濾、欄位排序」的現代化 DataGrid 列表。
  * 包含：系統交易總表、玩家(Guest)餘額速查表、機台參數列表。
* **手動機台狀態**
  * 沒有 1 分鐘心跳監控燈號。機台清單會顯示「最後一次同步帳單的時間」，供管理員人工判斷該機台是否活躍。

### 2.3 單機本地後台 (Local Backend) (預估工時: 18 MD)
* **視覺化本機設定介面 (Local Settings UI)**
  * **完全抹除修改 `.env` 的需求**。機台管理者能在本機開啟專屬設定網頁，透過 UI 介面設定：「中央 API IP 位置、機台綁定序號」。
* **現場安全開洗分通道**
  * 提供專屬的輸入介面，並以 PIN 碼保護，防止玩家誤觸。開洗分完成後會顯示「已排入同步佇列」。

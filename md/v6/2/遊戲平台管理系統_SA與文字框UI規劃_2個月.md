## 系統架構 (SA) 與 UI 文字框設計詳細規劃

## 1. 系統架構詳細規劃 (基礎商用版)

### 1.1 具備斷線保單的資料庫架構
* **上傳與重試機制**：
  * 當機台按下「開分/洗分」，本地 SQLite 的 `local_transactions` 新增一筆紀錄，並建立一筆 `Sync_Job`。
  * 背景排程 (Cron/Worker) 每隔 3 分鐘將未完成的 `Sync_Job` 掃出，打包 (Batch) 推送給中央 API。
  * 若傳送失敗 (如 Timeout)，本地不會拋棄資料，而是延遲至下一個 3 分鐘再次嘗試。這確保了「拔網路線依舊能營業，插上網路線自動結算」。
  * ** Append-only**：資料僅為「新增」，嚴格禁止 UPDATE 金額，避免複雜度。

### 1.2 無串接 API 與扁平化遊戲介接
* **無串接**：本期架構不包含第三方遊戲錢包邏輯。
* **本地連動**：遊戲端僅透過簡單的本地 HTTP GET/POST 與 SQLite 同一台機器上的 Node.js 服務索取目前餘額，並在遊戲結束時報告輸贏。

### 1.3 靜態機台管理 (無心跳)
* 機台不會發送無意義的 `/ping` 來浪費伺服器資源與開發時間。
* 中央伺服器單純透過每一台機器「最後上傳有效訂單的時間 (`last_sync_time`)」來推測機台是否營業。

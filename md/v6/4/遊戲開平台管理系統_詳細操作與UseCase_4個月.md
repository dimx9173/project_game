# 遊戲平台管理系統 (MVP Phase 1) - 方案 A：企業旗艦版 (4個月)
## 詳細操作流程與 Use Case 說明

本方案主打「規模化營運的自動化維修與對帳」，以下這三個 Use Case 凸顯了旗艦版與其他版本的巨大差距：在幾十台甚至上百台的營運規模中，省去了大量的工程師救援與財會核算時間。

---

## Use Case 1: 老闆喝咖啡一鍵發布的全省熱更新 (OTA Deployment)
**【情境說明】** 你有一百台拉霸機放在全台不同的角落。今天博弈工程師調整了老虎機的中獎機率演算法 (RTP)。如果是在 基礎商用版，你需要派六個工程師坐高鐵去插 USB 更新。在 企業版中，這是全自動的。
**【主要角色】** 總部營運工程師、各地一百台機台

1. 總部工程師打包了一份 `GameApp_Update_1.2.zip` 檔案。
2. 他打開中央管理後台，進入 `[ OTA 版本發布中心 ]`。
3. 把壓縮檔拖曳進系統。填入版本號 `v1.2.0`。按下發布。
4. 全球 100 台機器中，目前有 80 台店面開機營業，有 20 台正在停電/關機。
5. 這 80 台開機的機器，在 1 分鐘內收到了中央發出的 `update_available` 訊號。
6. 這 80 台機器中，有 50 台目前有客人在玩。這 50 台將這個更新任務 **加入待辦事項 (Pending)** 不打擾客人。剩下 30 台發現閒置超過 5 分鐘，立刻跳出提示窗，花 10 秒鐘下載、自己關閉自己、解壓縮替換檔案、重新把軟體開起來。
7. 兩小時後，客人下機了。原先等待中的 50 台機器偵測到閒置，自動接力完成重啟與升級。
8. 隔天早上，停電的 20 台機器插上電。連上網路的首 1 分鐘，立刻下載更新並重裝為新版本。
9. 總部工程師從頭到尾都在冷氣房看著螢幕上的進度條變成 100%。

## Use Case 2: 金流爭議處理與會計沖銷 (Financial Reconciliation)
**【情境說明】** 客人投了 $5,000。此時發生罕見的斷網加上硬碟瞬間滿載。客人說 App 當掉而錢被扣了，但店家看總表發現沒有這筆營收，發生客訴爭議。
**【主要角色】** 現場店長、總部財務、中央伺服器

1. 客人發生命理級的機率報錯，單機端沒顯示餘額增加。客人找店長吵架。
2. 店長無法處理，回報給總部財務。
3. 總部財務進入 企業版專屬的 `[爭議對帳處理專區]`。由於 企業版的非同步 Queue 會盡全力在上傳失敗時紀錄完整的 Request Header 與 Error Log。
4. 財務在報表中看到一筆 14:02 發生的**紅字異常交易**: `$5,000 (狀態：遺失封包/未寫入)`。
5. 財務確認了此單據存在過，點選這筆報單旁邊的 **[人工沖銷]** 按鈕。
6. 後台系統不會強制覆寫原始單據（破壞會計原則），而是由系統自動新增一筆名為 `SYS_Audit` 的 `- $5,000` 帳務去把這筆斷腳的紀錄撫平。並發送一根 API 強制讓那台機器加上 $5,000 的額度。
7. 客人的機器畫面上出現 `$5,000`，立刻安撫了客訴。且財報完全符合查帳邏輯。

## Use Case 3: 泛用型的第三方錢包中心轉入轉出 (Payment Router)
**【情境說明】** 今天 V6 平台內已經掛了 PG 洗分機、JDB 滿貫大亨、跟某家真人百家樂。玩家要帶著一萬元暢遊這三個不同公司的遊戲。
**【主要角色】** 玩家、中央 Payment Router

1. 玩家坐下來，現場人員幫他開了 `$10,000` 分進入他此機台的「中心錢包」。
2. 玩家點擊畫面的 PG 洗分機。
3. 此時，中央後台的 `PaymentRouter` 接收指令。這是個工廠模式 (Factory Pattern) 抽象層。它實作了 `pg_adapter.js`，將 `$10,000` 扣下，轉化為 PG 的 API 格式，成功讓玩家進去玩。
4. 玩家玩了十分鐘剩 `$2,000`。覺得手氣差。退出 PG。
5. V6 扣除 PG 餘額，將 `$2,000` 退回中心錢包顯示。
6. 玩家點擊 JDB 滿貫大亨。
7. `PaymentRouter` 換成調用 `jdb_adapter.js`，無縫串接，將 `$2,000` 丟進新遊戲。
8. 玩家爽快遊玩。
9. 對於未來的工程師來說，如果老闆下令「下週我們要引進第四間公司的遊戲」，工程師完全不需要動到 V6 關於這台機器的結算邏輯或主架構。他只要去 `adapters/` 資料夾中新增一隻 `new_vendor.js` 的轉換小檔案即可。這就是 企業版系統具備「未來擴展性」的最大價值展現。

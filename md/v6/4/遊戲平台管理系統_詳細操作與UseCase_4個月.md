# 遊戲平台管理系統 (MVP Phase 1) - 方案 A：完整基礎版 (4個月/80w)
## 詳細操作流程與 Use Case 說明

本文件旨在具象化 4 個月 MVP 版本的「使用者實際上會怎麼操作」。該版本因為功能較為完整，涉及多層級角色管理、非同步斷線容錯以及遠端派發更新，故我們定義了以下三個核心 Use Case。

---

## Use Case 1: 網管營運中斷與離線容錯機制 (Offline Recovery)
**【情境說明】** 現場網路突然中斷 (如中華電信斷線半小時)，但現場客人仍在遊玩。
**【主要角色】** 現場操作員、機台端背景 Worker、中央 PostgreSQL

### 階段一：斷線發生與本地紀錄
1. 機台端畫面不會跳出任何阻擋。操作員依然可以對指定的玩家輸入「開分 $1,000」。
2. 系統將 `$1,000` 寫入本地 SQLite `transactions` 表，玩家立刻可以看到遊戲畫面內的錢包增加，開始遊玩。
3. 同時，系統將此異動封裝寫入本地 SQLite `task_queue` 表，狀態標記為 `pending`。
4. 背景 Worker 每 30 秒嘗試連線中央 API。發現 HTTP 狀態碼為 503 或 API Timeout，中斷本次嘗試，`task_queue` 停留於 `pending`，但機台**右上角文字框由 `[網路狀態: 🟢 已連線中央]` 轉為 `[網路狀態: 🔴 離線 (佇列積壓: 1 筆)]`**。

### 階段二：網路復原與同步
1. 半小時後網路恢復。背景 Worker 偵測到 API `/health` 回傳 200 OK。
2. Worker 將過去半小時內積壓的 50 筆 `pending` 紀錄以 JSON Array 格式打包 (Batch) 推送給中央。
3. 中央收到並依序寫入 PostgreSQL，回傳 HTTP 200。
4. 機台端收到確認後，將此 50 筆紀錄狀態更新為 `completed`，清空本地佇列警報，恢復綠燈。

## Use Case 2: 遠端版本派送 (OTA Updates)
**【情境說明】** 系統有了新功能或查殺 Bug，代理商不希望人力跑到每一台實體機台前插 USB 更新，而是由中央派送。
**【主要角色】** 超級管理員、機台端 OTA 背景服務

1. **上傳發布套件**：
   管理員登入集中式後台，進入「OTA 版本發布」，將最新的 `.zip` 壓縮檔上傳至系統 (底層放到直連 S3)。並填寫版本號 `v1.2.5` 與更新日誌。
   ```text
   UI 畫面：
   [ 新增發布版本 ]
   檔案上傳: [app_v1.2.5.zip]  強制更新: [v]
   描述: 修正吃角子老虎機顯示 Bug。
   [確定發布]
   ```
2. **機台心跳觸發 (Heartbeat)**：
   目標機台（目前版本為 `v1.2.4`）在背景每 5 分鐘打一次 Heartbeat 給中央。
   中央分析比對後，在 Response 的 Payload 中回傳：`{"update_available": true, "url": "...", "md5": "...", "force": true}`。
3. **機台背景下載與重啟**：
   機台端遊戲畫面依然維持給玩家玩。
   背景偷偷下載 zip 檔案。下載完畢且 MD5 驗證無誤後。
   若判定機台目前閒置 (Idle)，則立刻觸發 `update.bat` 關閉現有主程式，解壓縮覆蓋檔案，自動重啟 App 進入 `v1.2.5`。若有玩家遊玩中，則等待至「結帳洗分完畢」的空檔才重啟。

## Use Case 3: 階層代理商與機台綁定 (RBAC Hierarchy)
**【情境說明】** 平台上有多個代理商加盟，每個代理商底下只能管自己的機台，超級管理員要能掌控全局。
**【主要角色】** 超級管理員、代理商 A、代理商 B

1. **建立代理商帳號**：
   超級管理員在後台選擇「新增代理商」，建立帳號密碼並派發角色為 `Agent`。
2. **分派機台控制權**：
   管理員到「機台清單」勾選 MAC 地址 `MAC-001` 與 `MAC-002`，指定其 Owner 為 `代理商 A`。
3. **資料隔離**：
   代理商 A 登入後台時，API Middleware 會自動附加上 `WHERE owner_id = 'AgentA_ID'` 的查詢條件。
   因此，代理商 A 的 Dashboard 總營收只會出現這兩台機器的數字，完全看不到競爭對手代理商 B 的資料。
   代理商 A 可以自行在自己名下創建低階的「現場操作員」帳號，該操作員只能幫這兩台機器進行查帳。

---
**【總結】**：本 4 個月方案提供了高強度的自動化，大大降低了「人工維修」的頻率，適合已經準備大規模展店鋪機台的維運情境。

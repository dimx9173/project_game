## 1. 文件概述

本文件詳細闡述遊戲平台管理系統的整體系統架構，包含集中式後台與單機本地後台兩個層次的架構設計、資料流、通訊機制等內容。

## 2. 系統架構概述

### 2.1 雙層後台架構

遊戲平台採用雙層後台架構設計。上層是集中式後台，負責全局資料監控、遠端機台管理、全平台交易統計、玩家集中管理等功能。下層是單機本地後台，負責本地機台管理、本地遊戲配置、本地交易結算、本地統計分析等功能。兩個後台相互獨立，但在連線時可進行資料同步與對帳。

### 2.2 系統組件

系統包含以下主要組件：集中式後台前端（React + Tailwind CSS），集中式後台後端（Express.js + Node.js），單機本地後台前端（React + Tailwind CSS），單機本地後台後端（Golang），集中式資料庫（PostgreSQL），單機本地資料庫（SQLCipher ），遊戲引擎，玩家端應用程式。

## 3. 集中式後台架構

### 3.1 前端架構

集中式後台前端採用 React 19 + Tailwind CSS 4。使用 Wouter
進行路由管理，shadcn/ui 提供統一的 UI 元件，Recharts
用於資料視覺化，React Context
進行狀態管理。支援響應式設計，適配桌面、平板、手機等多種裝置。

### 3.2 後端架構

集中式後台後端採用 Express.js 框架，Node.js 執行環境。提供 RESTful API
介面，支援 WebSocket 實時通訊。使用 PostgreSQL 資料庫進行資料持久化，Redis
進行快取加速。

### 3.3 核心模組

集中式後台包含 10 個核心模組：儀表板模組（全平台 KPI 監控），機台管理模組（遠端機台管理），遊戲管理模組（全局遊戲庫管理），玩家管理模組（集中式玩家管理），錢包交易模組（全平台交易統計），串接遊戲管理模組（遊戲商 API 管理），多人遊戲模組（多人遊戲管理），使用者權限模組（帳號與角色管理），版本更新模組（應用程式與單機本地後台主程式 OTA 遠端更新管理），介面設定模組（玩家端配置管理）。

## 4. 單機本地後台架構

### 4.1 前端架構

單機本地後台前端採用與集中式後台相同的（React 19 + Tailwind CSS
4），確保設計風格與使用體驗的一致性。支援本地化部署，可在機台上直接運行。

### 4.2 後端架構

單機本地後台後端採用 Golang 語言開發，編譯為二進位執行檔以防止程式碼遭遇逆向工程或竄改。提供 RESTful API 介面，支援 WebSocket 實時通訊。使用 SQLCipher (加密版 SQLite) 進行本地資料儲存，並結合 Server 核發的 Secret Salt 與硬體指紋動態生成資料庫解密金鑰，防止實體機台資料外洩。支援本地快取。

### 4.3 核心模組

單機本地後台包含 9 個核心模組：儀表板模組（本機 KPI
監控），機台管理模組（本地機台管理），遊戲管理模組（本地遊戲庫管理），交易管理模組（本地交易統計），玩家管理模組（本地玩家管理），統計報表模組（本地統計分析），系統設定模組（本地配置管理），同步與對帳模組（資料同步驗證），監控與日誌模組（系統監控與日誌管理）。

## 5. 資料同步與對帳機制

### 5.1 同步流程

當單機連線到集中式後台時，自動觸發資料同步流程。首先檢查離線期間的本地變更，上傳本地交易紀錄到後台。然後下載後台的遊戲更新、配置更新。最後驗證資料一致性，完成同步。

### 5.2 衝突解決

當機台離線期間進行的操作與後台操作衝突時，系統採用時間戳優先、版本號優先、人工審核、交易回滾等策略進行解決，確保資料最終一致。

### 5.3 對帳機制

系統定期進行對帳檢查，驗證本地資料與後台資料的一致性。對於不一致的項目，記錄並標記為待審核，由管理員手動處理。

## 6. 安全性架構

### 6.1 認證與授權

系統採用帳號密碼認證，使用 JWT Token
進行會話管理。實施基於角色的存取控制（RBAC），不同角色的使用者擁有不同的操作權限。

### 6.2 資料加密

所有敏感資料（玩家帳號、交易紀錄、配置資訊）使用 AES-256
加密演算法進行加密儲存。通訊採用 HTTPS 加密協議，確保資料傳輸安全。

### 6.3 交易簽名驗證

所有交易使用 RSA
數位簽名進行驗證，確保交易的真實性與完整性，防止交易被竄改。

### 6.4 稽核日誌

記錄所有重要操作的詳細日誌，包括操作者、操作時間、操作內容、操作結果等資訊，支援追蹤與審計。

### 6.5 機台硬體特徵綁定認證與防竄改機制

為確保連線到集中式後台的機台為合法授權，系統實施「硬體指紋綁定 (Device Fingerprinting)」認證與離線防護機制。為解決機台「離線悖論 (Offline Paradox)」，本地端程式 (Golang) 的防竄改與加密金鑰衍生流程分為以下三階段：

1.  **階段一：註冊與初始化 (僅執行一次)**
    - 新機台初次連線時，本地後台將擷取多個硬體特徵（如主機板序號、CPU ID、MAC 位址等）並經 SHA-256 運算產生獨一無二的「**硬體指紋 (Machine Fingerprint)**」。
    - 由管理員於集中式後台人工審核綁定後，集中式後台核發專屬的長效 Token 以及一組高強度的「**機密鹽值 (Secret Salt)**」，存放在本地機台設定檔中。
2.  **階段二：金鑰動態衍生與解鎖 (每次啟動時)**
    - 後續機台**每次啟動（含離線斷網狀態）**，本地端 Golang 程式皆會即時重新讀取當下的硬體特徵並生成硬體指紋。
    - 透過密碼學派生函數 (如 PBKDF2 或 Argon2)，將「**當前硬體指紋**」結合本地儲存的「**Secret Salt**」，動態衍生出 256-bit 的 **SQLCipher 資料庫解密金鑰**。
    - 此設計確保若硬碟被拔走或資料被複製到其他電腦，因硬體指紋不符，將絕對無法解開且讀取本地資料庫，保障玩家餘額與交易紀錄安全。
3.  **階段三：日常營運防護與雙向安全連線**
    - 解鎖資料庫後，所有的 SQL 操作皆由 SQLCipher 在背景即時加解密。斷電時金鑰即刻從 RAM 中消失，恢復加密狀態。
    - 在連線狀態下發送 API 請求時，皆需攜帶 Token 並與硬體指紋進行交叉驗證，且本地 API 封包需利用私鑰進行 RSA 簽章 (Transaction Signing)，防止身份偽造或交易竄改。若遇異常，管理員可「一鍵註銷」撤銷該機台連線權限。

## 7. 系統升級與 OTA 更新架構

為解決分散於各實體機台的本地後台程式更新耗時費力的痛點，系統內置一套高可靠的「單機本地後台遠端 OTA (Over-The-Air) 升級機制」。

### 7.1 OTA 更新流程設計

1.  **發佈更新包 (Publish)：** 平台管理員登入集中式後台，上傳新版單機本地後台的 Golang 執行二進位檔 (或壓縮包)、版本號描述與更新說明，並產生該檔案的 SHA-256 數位簽章與雜湊值。
2.  **派發任務 (Dispatch)：** 管理員可選擇對「全平台」、「特定群組」或「指定機台」派送 OTA 更新任務，並可設定為「強制更新」或「閒置時更新」。
3.  **機台輪詢與下載 (Poll & Download)：** 單機本地後台透過 WebSocket 或定期輪詢 API 得知有新版本後，啟動背景下載機制 (Background Download)，將更新包下載至本地暫存區（如 `/tmp/updates`）。
4.  **校驗與備份 (Verify & Backup)：**
    - **完整性校驗：** 下載完成後，比對檔案的 SHA-256 雜湊值與數位簽章，確保更新包在傳輸過程中未損壞且未被竄改。
    - **安全備份：** 校驗通過後，本地端程式先將現有的舊版執行檔與資料庫進行完整備份（含舊版的解密還原機制）。
5.  **覆蓋與重啟 (Deploy & Reboot)：**
    - 本地後台自動關閉當前的應用服務層 (Graceful Shutdown)。
    - 啟動一支獨立且輕量的 Update Agent (或由系統 Shell 腳本接管)，將新版二進位檔覆蓋舊檔。
    - 重啟主程式進程。
6.  **健康檢查 (Health Check)：** 新版主程式啟動後，執行自我健康檢查（例如檢查金鑰衍生與資料庫連線是否正常、與集中式後台建立心跳連線）。
7.  **狀態回報 (Report)：**
    - **成功：** 向集中式後台回報「更新成功」及最新版本號。
    - **失敗 (回滾機制)：** 若健康檢查超時或啟動失敗，Update Agent 自動中斷新程式，還原方才備份的舊版執行檔與 DB，並回報「更新失敗與錯誤代碼」，確保實體機台的 **高可用性與零停機風險 (Zero Downtime Risk)**。

## 8. 效能與可擴展性設計

### 7.1 效能優化

集中式後台使用 Redis
進行快取加速，減少資料庫查詢次數。使用資料庫索引優化查詢效能。使用 CDN
加速靜態資源傳輸。支援 API 請求的非同步處理，提高系統吞吐量。

### 7.2 可擴展性設計

集中式後台採用微服務架構，支援水平擴展。資料庫支援分片，實現資料層的水平擴展。API
支援版本管理，支援向後相容。支援第三方遊戲商接入，實現業務層的擴展。

## 9. 部署架構

### 9.1 集中式後台部署

集中式後台部署在雲端或資料中心。前端靜態資源部署在
CDN。後端服務使用容器化部署（Docker），支援自動擴展。資料庫使用主從複製實現高可用。

### 9.2 單機本地後台部署

單機本地後台與遊戲引擎部署在同一台機器上。支援 Windows、Linux、macOS 等作業系統。編譯為獨立執行的二進位檔 (Golang)，使用 SQLCipher 加密本地資料庫。

### 9.3 網路拓撲

集中式後台與單機本地後台通過 Internet
連線。單機本地後台與遊戲引擎通過本地網路連線。玩家端應用程式與遊戲引擎通過本地網路連線。

# 遊戲平台管理系統：SA 系統架構 V5 與 V8 版本差異分析

## 1. 技術棧與框架 (Tech Stack Overhaul)
這是本次升級最核心的結構性變動。

| 項目 | V5.0 整合版 | V8.0 旗艦版 |
| :--- | :--- | :--- |
| **後端框架** | Node.js (Express.js) | **C# .NET 8 (ASP.NET Core)** |
| **前端框架** | React 19 + Tailwind CSS | **Blazor Server + Radzen Blazor** |
| **主資料庫** | MySQL | **PostgreSQL 15+** |
| **即時通訊** | WebSocket | **SignalR** |
| **ORM** | (未指定，推測為 Sequelize/Prisma) | **Entity Framework Core** |

---

## 2. 系統功能擴展與深化

### 2.1 中央後台功能強化
*   **OTA (Over-The-Air) 服務 (旗艦新增)**：V8 加入了完整的遠端包派發機制，包含版本控制、MD5 比對、與熱更新指令發送。
*   **深層監控系統 (新增)**：從 V5 的簡單狀態顯示，進化為監控機台硬體細節（CPU 負載、RAM 佔用、網路延遲圖表）。
*   **爭議帳務處理 (旗艦新增)**：V8 建立了「爭議對帳處理專區」，專門處理斷線異常、人工核銷與會計審計邏輯。
*   **代理商樹狀架構 (新增)**：明確定義了三層代理架構 (Admin > Agent > Operator) 的分潤與權限繼承。

### 2.2 單機本地後台強化
*   **非同步交易佇列**：V8 引入了具備 **Exponential Backoff (指數退避)** 機制的傳輸引擎，加強了在極端惡劣網路環境下的資料上傳可靠性。
*   **背景更新接收器**：單機端新增了靜默下載更新包的功能，確保不干擾現場玩家。

---

## 3. 資料結構與安全性變更

*   **資料庫選型**：由通用型的 MySQL 遷移至更適合處理複雜 JSON 欄位與高併發寫入的 **PostgreSQL**。
*   **安全加密標準**：由 V5 的 AES-256 升級為更安全的 **AES-256-GCM** 認證加密模式。
*   **認證機制**：中央後台維持 JWT，但單機本地端明確採用 **PIN Code** 快速登入設計，符合現場操作情境。

---

## 4. 衝突與對帳策略 (Reconciliation Policy)
V8 在 SA 文件中詳細定義了「邊界協定」與「衝突解決方針」：
*   **中央優先原則**：涉及交易結果時以中央資料庫為基準。
*   **本地保留原則**：所有本地操作日誌強制保留，作為人工審核爭議帳務的依據，「絕不憑空銷毀任何記帳斷層」。

---

## 5. 總結
V5 設計適合快速開發與通用部署，而 **V8 則是專為「大規模、高價值交易、遠端分散式場景」量身打造的旗艦規格**。技術棧的轉向 (.NET 8) 與隨之而來的型別安全、長連線穩定性提升，是支撐 V8 各項高級監控與自動化功能的基石。

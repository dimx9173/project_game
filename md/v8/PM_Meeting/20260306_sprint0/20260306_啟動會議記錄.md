2. 專案背景與開發動機 針對目前客戶在營運多台遊戲機台時面臨的核心管理困境，本專案旨在解決現有產品的技術瓶頸：

* 分散式管理缺口： 缺乏能有效媒合硬體實體並實現遠端分散式管理的整合系統。
* 對帳邏輯混淆： 現有產品介面雖美，但數據一致性不足，產生帳務偏差時難以追溯。
* 維護成本極高： 系統更新需依賴人力親臨現場使用隨身碟作業，且需打開機台外殼，極度耗時耗力。
* 網路環境挑戰： 需在不穩定網路環境下，確保中央與地方數據同步及敏感操作的授權穩定性。

3. 三大核心系統功能 (Core Features)

* OTA 無感更新： 核心目標在於實現遠端軟體更新，消除「打開機台」的人力維護需求。系統需支援自動派發、校驗與 回滾 (Rollback) 機制，確保更新過程不干擾機台營運。
* 帳戶對帳系統： 採用非同步交易處理架構。系統需自動攔截中央與本地端的餘額偏差，並將爭議交易移至「爭議專區」。該設計並非單純攔截，而是提供清晰的對照數據與可追溯性，以利管理人員進行手動裁決與結案。
* 動態授權與資安： 針對「開分」與「洗分/兌換」等敏感性金流操作，建立中央授權機制。嚴格執行「無記錄即無交易」原則，確保每筆動作皆有日誌可循。

4. 技術架構與開發設計規格

* 後端技術棧： 系統由 V6 轉至 V8/B 版本後，後端統一採用 C# (.NET) 開發。此決策是基於客戶端（CSP）之現有維護人員與應用程式使用者皆熟悉 C# 資源，利於後續維護與資安對接。
* API 規範： 統一使用 OpenAPI (Swagger) 格式。區分為「UI 串接 API」（管理介面用）與「機台通訊 API」（硬體通訊用）。
* 功能矩陣與技術債處理：
  * 系統核心由「功能需求矩陣」驅動，每一功能項均分配唯一代碼 (Code)，作為資料庫設計、SA/SD 文件及流程圖（開分、下注、同步流等）的索引基礎。
  * 技術債預警： 目前功能矩陣規模已達 2000-3000 行，為避免文檔過大成為開發瓶頸，近期將進行文檔重構 (Refactoring) 以拆分子檔案。
* 監控機制： 導入 Heartbeat (HB) 心跳機制，專門用於監控機台硬體狀態與詳盡的操作日誌 (Operation Logs)。

5. 介面原型 (Mockup) 與 UI/UX 規劃 本專案區分「本機端」與「雲端」兩套介面，其設計重點如下：

| 比較項目 | 本機端 (Local Machine)                                    | 雲端管理後台 (Cloud Dashboard)                               |
| -------- | --------------------------------------------------------- | ------------------------------------------------------------ |
| 介面比例 | 長條型畫面（手機比例）                                    | 橫式 Web 介面                                                |
| 操作邏輯 | 無鍵盤設計，以大型觸控按鈕為主                            | 滑鼠/鍵盤操作，側邊選單導覽                                  |
| 核心功能 | 連線設定（IP/Server）、同步狀態、OTA 進度、機台硬體資訊。 | 儀表板總覽、機台遠端控制、詳細交易與營收報表、OTA 版本控管。 |
| 狀態監控 | 等待傳送 (Pending Upload)狀態紀錄                         | 操作日誌查詢、監控中心、異常帳務處理。                       |
| 權限管理 | 單機操作權限設定                                          | 角色 (Role) 與帳號 (Account) 分離，支援自訂職位權限。        |

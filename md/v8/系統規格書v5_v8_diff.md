# 遊戲平台管理系統：需求規格書 (SRS) V5 與 V8 版本差異分析

## 1. 核心技術架構升級
V8 版本進行了全面的技術棧遷移，從原本的 JavaScript 生態系轉向企業級的 .NET 生態系，大幅提升了系統的嚴謹性與可維護性。

| 項目 | V5.0 整合版 | V8.0 旗艦版 |
| :--- | :--- | :--- |
| **技術堆疊** | React / Node.js / Express | **C# .NET 8 / Blazor Server** |
| **主要資料庫** | MySQL | **PostgreSQL 15+** |
| **本地資料庫** | SQLite / MySQL | **SQLite (標準化)** |
| **快取機制** | Redis | **Redis 7+ (高效能版)** |
| **即時通訊** | WebSocket | **SignalR (整合回連機制)** |

---

## 2. 集中式後台功能演進 (Centralized Backend)

### 2.1 監控與管理深度
*   **機台監控**：V5 僅有基礎狀態；V8 提升至「旗艦級監控」，包含 **1 分鐘頻率心跳、CPU/RAM 資源監測圖表**，用於預防性維護。
*   **OTA 服務**：V8 建立了「全球 OTA 無感熱更新發布中心」，具備 MD5 雜湊審查、排程更新與失敗自動還原機制。

### 2.2 帳務與報表
*   **報表維度**：V5 為基礎統計；V8 具備「多維度交叉報表」，可針對多帳號、多機台進行立體營運曲線比對。
*   **爭議對帳**：V8 新增了「爭議對帳處理專區」，專為網路異常導致的「中央/單機不一致」提供會計審核與人工核銷路徑。

### 2.3 擴充性設計
*   **遊戲商串接**：V8 採用標準化的遊戲商 API 介面，使第三方遊戲商的串接標準化；V5 則是較為耦合的單一串接。
*   **權限系統**：V8 升級為「動態 RBAC」，可針對上百個 API 進行細粒度的權限勾選。

---

## 3. 單機本地後台功能演進 (Local Backend)

*   **資料傳輸**：V8 引入了 **Exponential Backoff (指數退避)** 與補償機制，強化離線後重新連線的資料恢復能力。
*   **系統設定**：V8 詳細化了本地設定，包含時區、備份與恢復、網路 Proxy 設置以及雙面驗證 (mTLS) 管理。
*   **熱更新接收**：單機端新增接收器模組，支援背景靜默接收 OTA 封包後自我重啟，不干擾現場營業。

---

## 4. 業務規則 (Business Rules) 與安全性

V8 在規格書中首次明確化了多項業務規則 (BR-XXX)：
*   **交易原子性**：確保餘額更新與記錄在同一個資料庫事務中。
*   **安全防護**：升級至 **AES-256-GCM** 加密，並強化了 HMAC-SHA256 交易簽名。
*   **防綁架設定**：引入與硬體特徵綁定的安全性檢測。

---

## 5. 遺漏或調整建議
在升級至 V8 旗艦版時，請特別注意：
*   **大廳配置**：V8 強調營運邏輯，但 V5 中較為直覺的「玩家大廳背景與排序」視覺配置功能在 V8 中被移動至「前端版面廣播模組」，需確認介面是否依然對操作員友善。
*   **測試模式**：確認 V8 的單機測試模式依然保留「模擬下注」功能，這對於現場工程師排除硬體故障至關重要。

---

## 6. 結論
V8 版本不只是版本號的跳躍，而是系統本質從「小型營運管理」向「**大型分散式、高安全性企業平台**」的轉型。C# .NET 8 的引入為未來 3-5 年的長期穩定運作奠定了堅實基礎。

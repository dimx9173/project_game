## 3. 心跳間隔一致性 ⚠️

| 文件 | 描述 | 狀態 |
| ---- | ---- | ---- |
| 規格書 | 「預設 30 秒，不得超過 60 秒」、「超過 3 分鐘無 Heartbeat 視為離線」 | ✅ |
| SA 文件 | 「定時 Heartbeat」、「機台心跳深度監測模組」 | ⚠️ |
| 資料庫設計 | 未明確說明間隔 | ❌ |
| SD 文件 | 「機台每分鐘發送心跳」（60 秒）、「超過 3 分鐘無心跳 → 離線」 | ⚠️ |
| UI 設計 | 「Heartbeat 心跳監控」 | ⚠️ |
| 功能矩陣 | 「機台每分鐘發送心跳」（60 秒）、「超過 3 分鐘無心跳視為離線」 | ⚠️ |
| API | `/machines/{id}/heartbeat` 未說明間隔要求 | ⚠️ |

**發現問題**:
- **規格書**: 預設 30 秒，上限 60 秒
- **SD 文件**: 每分鐘（60 秒）
- **功能矩陣**: 每分鐘（60 秒）

**評估**: ⚠️ **輕微不一致**。規格書的「預設 30 秒」與其他文件的「60 秒」不同，但「3 分鐘離線判定」一致。

**建議**: 統一表述為「預設 60 秒，可配置 10-300 秒」，或明確說明 30 秒為建議值、60 秒為最大值。

## 4. 交易簽名演算法一致性 ⚠️

| 文件 | 描述 | 狀態 |
| ---- | ---- | ---- |
| 規格書 | 「AES-256-GCM + RSA」、「交易簽名驗證（HMAC-SHA256）」 | ⚠️ |
| SA 文件 | 「所有交易使用 RSA 數位簽名進行驗證」 | ✅ |
| 資料庫設計 | 「敏感欄位（密碼、交易簽名）使用 AES-256-GCM 加密」 | ⚠️ |
| SD 文件 | 「AES-256-GCM + RSA 加密」、「HMAC-SHA256 資料完整性」、「RSA-2048 API 簽名」 | ✅ |
| UI 設計 | 「交易紀錄已建立（含簽名）」 | ⚠️ |
| 功能矩陣 | 未說明 | ❌ |
| API | 未說明 | ❌ |

**發現問題**:
- **規格書**: 同時提到 RSA 和 HMAC-SHA256，但未說明使用場景
- **SD 文件**: 清楚說明「HMAC-SHA256 用於資料完整性」、「RSA-2048 用於遊戲商 API 簽名」
- **資料庫設計**: 提到 AES-256-GCM 加密簽名欄位，但未說明簽名演算法

**評估**: ⚠️ **表述不夠清晰**。SD 文件最詳細，規格書需要補充說明。

**建議**: 
- 規格書補充：「交易簽名使用 RSA-2048，資料完整性使用 HMAC-SHA256」
- 資料庫設計補充：「signature 欄位儲存 RSA 簽名（AES-256-GCM 加密）」

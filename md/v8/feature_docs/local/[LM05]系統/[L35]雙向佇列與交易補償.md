# [L35] 雙向佇列與交易補償

**功能代碼**: L35  
**所屬模組**: [M05]系統  
**最後更新**: 2026-03-09

---

## 1. 功能說明

雙向佇列機制是本地與中央資料同步的核心架構，旨在解耦本地關鍵操作與遠端 API 調用。透過佇列，機台即使在斷網期間也能正常開洗分、執行遊戲，並在恢復連線後自動補償資料。

### 技術機制：
- **下行佇列 (Downlink Queue)**：存儲來自中央的設定更新、遊戲列表、版本通知。
- **上行佇列 (Uplink Queue)**：存儲本地生成的交易紀錄、操作日誌、機台快照。
- **交易補償 (Compensation)**：若同步失敗，系統會記錄失敗狀態並在下次重試中執行補償邏輯。

---

## 2. 操作流程

### 2.1 資料生產與排隊 (Production)
1. **觸發操作**：管理者或玩家執行開分、洗分或調整設定。
2. **寫入本地庫**：系統優先更新本地資料庫以確保操作流暢。
3. **加入佇列**：同時將該操作打包為 JSON 交易封裝，存入 `sync_queue` 資料表，狀態標記為 `PENDING`。

### 2.2 背景同步與確認 (Synchronization)
1. **背景掃描**：同步服務週期性（如每 30 秒）或由連線事件觸發掃描 `sync_queue`。
2. **批次上傳**：將 `PENDING` 的交易記錄按批次（如 100 筆/批）發送至中央伺服器。
3. **伺服器回應**：
   - **成功 (200 OK)**：本地接收回復後，將佇列項標記為 `COMPLETED` 並清理。
   - **失敗 (4xx/5xx)**：將狀態更新為 `FAILED`，並觸發 [L36] 指數退避重試。

---

## 3. 屬性/佇列欄位

| 欄位名稱     | 說明                             | 類型     | 設定值範例           |
| ------------ | -------------------------------- | -------- | -------------------- |
| 佇列方向     | 資料流向                         | Enum     | UP (上傳) / DOWN (下載)|
| 資料類型     | 封包內容分類                     | Enum     | TRANSACTION / LOG    |
| 任務狀態     | 目前同步進度                     | Status   | PENDING / FAILED     |
| 重試次數     | 該項任務已嘗試同步的次數         | Integer  | 0 (初次) - 5 (上限)  |
| 錯誤日誌     | 上次同步失敗的詳細錯誤訊息       | Text     | Timeout / Auth Error |

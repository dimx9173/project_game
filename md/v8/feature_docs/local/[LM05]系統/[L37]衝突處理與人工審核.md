# [L37] 衝突處理與人工審核

**功能代碼**: L37  
**所屬模組**: [M05]系統  
**最後更新**: 2026-03-09

---

## 1. 功能說明

在分布式架構中，當本地機台與中央伺服器針對同一物件（如設定參數、使用者餘額）存在不同的修改且版本不一致時，會觸發衝突處理機制（Conflict Resolution）。

### 處理原則：
- **版本號比對**：基於時間戳記或單調遞增版本號。
- **自動解決**：針對低風險資料採用預設策略。
- **人工審核**：針對高風險資產（如金額、關鍵日誌）掛起同步，等待管理員決定。

---

## 2. 操作流程

### 2.1 衝突檢測流程
1. **傳送請求**：本地提交同步封包，含目前的 `version_id`。
2. **中心驗證**：中央發現在資料庫中的 `version_id` 已被其他來源更新。
3. **返回衝突**：回傳 HTTP 409 Conflict 及最新的中央版本資料。

### 2.2 解決策略執行
1. **識別類型**：
   - **配置類**：執行「中央優先 (Central Wins)」，強制下載最新設定。
   - **交易類**：執行「本地優先 (Local Wins)」，嘗試將本地補償交易合併入中央日誌。
   - **資產類 (餘額)**：觸發「人工審核」。
2. **人工審核介意**：管理員在後台看到「待審核衝突」清單。
3. **對比與決策**：查看本地值、中央值及修改時間，選擇「接受本地」或「按中央為主」。
4. **恢復同步**：手動確認後，同步狀態恢復正常。

---

## 3. 衝突屬性

| 欄位名稱     | 說明                             | 類型     | 狀態範例             |
| ------------ | -------------------------------- | -------- | -------------------- |
| 衝突類型     | 發生衝突的模組                   | Enum     | BALANCE / CONFIG     |
| 本地版本號   | 機台端持有的最後版本             | String   | v2.3.1               |
| 中央版本號   | 伺服器端持有的現行版本           | String   | v2.4.0               |
| 解決策略     | 當前系統採用的自動或手動策略     | Enum     | MANUAL / AUTO_PATCH  |
| 處理狀態     | 該衝突項是否已排解               | Status   | PENDING / RESOLVED   |

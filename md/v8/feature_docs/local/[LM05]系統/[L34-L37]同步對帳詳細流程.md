# [L34-L37] 同步對帳詳細流程

**功能代碼**: [L34], [L35], [L36], [L37]  
**所屬模組**: [M05]系統  
**最後更新**: 2026-03-07  
**優先級**: 🔴 高

---

## 📋 概述

同步對帳機制確保本地機台與集中式後台的資料一致性，包含同步狀態檢視、雙向佇列、指數退避重試、以及衝突處理。

---

## 🔄 完整流程圖

### 雙向同步機制 (WebSocket + REST)

```
┌─────────────────────────────────────────────────────────────┐
│                    集中式後台 ↔ 本地機台                     │
│                                                             │
│  集中式 → 本地 (下行推送)                                   │
│  - WebSocket (SignalR) `CONFIG_UPDATE` 事件                 │
│  - 本地主動調用 `/api/v1/configs/{id}` 下載內容              │
│  - 套用成功後調用 `/api/v1/machines/{id}/config/status`      │
│                                                             │
│  本地 → 集中式 (上行同步)                                   │
│  - 交易紀錄上傳 (`POST /api/v1/sync/upload`)                 │
│  - 操作日誌上傳                                             │
│  - 機台狀態回報                                             │
└─────────────────────────────────────────────────────────────┘
```

### 雙向佇列機制

```
┌─────────────────────────────────────────────────────────────┐
│  本地機台                                                   │
│                                                             │
│  ┌─────────────────┐         ┌─────────────────┐           │
│  │  下行佇列        │         │  上行佇列        │           │
│  │  (Central→Local)│         │  (Local→Central)│           │
│  │                 │         │                 │           │
│  │ - 設定更新       │         │ - 交易紀錄       │           │
│  │ - 遊戲更新       │         │ - 操作日誌       │           │
│  │ - 版本通知       │         │ - 狀態回報       │           │
│  │                 │         │                 │           │
│  │ 狀態：待處理     │         │ 狀態：待上傳     │           │
│  └─────────────────┘         └─────────────────┘           │
│           ↓                           ↓                     │
│  處理下行任務                    連線時自動上傳              │
│  - 套用設定                     - 批次上傳                  │
│  - 更新遊戲                     - 確認機制                  │
│  - 準備更新                     - 失敗重試                  │
└─────────────────────────────────────────────────────────────┘
```

### 指數退避重試邏輯

```
┌─────────────────────────────────────────────────────────────┐
│  上傳失敗                                                   │
│     ↓                                                       │
│  第 1 次重試：立即 (退避係數：2^0 = 1)                       │
│     ↓ 失敗                                                  │
│  等待 10 秒                                                   │
│     ↓                                                       │
│  第 2 次重試：間隔 10 秒 (退避係數：2^1 = 2)                  │
│     ↓ 失敗                                                  │
│  等待 20 秒                                                   │
│     ↓                                                       │
│  第 3 次重試：間隔 20 秒 (退避係數：2^2 = 4)                  │
│     ↓ 失敗                                                  │
│  等待 40 秒                                                   │
│     ↓                                                       │
│  第 4 次重試：間隔 40 秒 (退避係數：2^3 = 8)                  │
│     ↓ 失敗                                                  │
│  等待 80 秒                                                   │
│     ↓                                                       │
│  第 5 次重試：間隔 80 秒 (退避係數：2^4 = 16)                 │
│     ↓ 失敗                                                  │
│  加入失敗佇列，通知管理員                                    │
│                                                             │
│  最大重試次數：5 次                                          │
│  初始間隔：10 秒                                             │
│  最大間隔：300 秒 (5 分鐘)                                    │
└─────────────────────────────────────────────────────────────┘
```

### 衝突檢測與解決

```
┌─────────────────────────────────────────────────────────────┐
│  檢測衝突                                                   │
│     ↓                                                       │
│  比對版本號/時間戳                                          │
│  - 本地版本 vs 中央版本                                     │
│  - 本地修改時間 vs 中央修改時間                             │
│     ↓                                                       │
│  發現衝突 (版本不一致)                                       │
│     ↓                                                       │
│  衝突解決策略                                               │
│                                                             │
│  策略 1: 中央優先 (Central Wins)                            │
│  - 適用於：系統設定、遊戲列表                               │
│  - 直接覆蓋本地資料                                         │
│  - 記錄衝突日誌                                             │
│                                                             │
│  策略 2: 本地優先 (Local Wins)                              │
│  - 適用於：交易紀錄、操作日誌                               │
│  - 保留本地資料                                             │
│  - 合併到中央                                               │
│                                                             │
│  策略 3: 人工審核 (Manual Review)                           │
│  - 適用於：重要設定、餘額資訊                               │
│  - 標記為「待審核」                                         │
│  - 暫停同步                                                 │
│  - 通知管理員介入                                           │
│     ↓                                                       │
│  記錄衝突解決結果                                           │
│  - 衝突類型                                                 │
│  - 解決策略                                                 │
│  - 最終結果                                                 │
└─────────────────────────────────────────────────────────────┘
```

### 人工審核流程

```
┌─────────────────────────────────────────────────────────────┐
│  系統檢測到需要人工審核的衝突                               │
│     ↓                                                       │
│  建立審核任務                                               │
│  - 記錄衝突詳情                                             │
│  - 標記為「待審核」                                         │
│  - 暫停相關同步                                             │
│     ↓                                                       │
│  通知管理員                                                 │
│  - 系統通知                                                 │
│  - Email 通知 (如設定)                                        │
│  - 顯示待審核清單                                           │
│     ↓                                                       │
│  管理員審核                                                 │
│  - 檢視衝突詳情                                             │
│  - 比較本地 vs 中央資料                                      │
│  - 選擇解決方案                                             │
│     ↓                                                       │
│  執行解決方案                                               │
│  - 接受本地版本                                             │
│  - 接受中央版本                                             │
│  - 手動合併                                                 │
│     ↓                                                       │
│  恢復同步                                                   │
│  - 更新同步狀態                                             │
│  - 繼續未完成的同步任務                                     │
│  - 記錄審核結果                                             │
└─────────────────────────────────────────────────────────────┘
```

## ⚠️ 異常處理

### 1. 網路中斷

**情境**: 本地機台與集中式後台網路中斷

**處理流程**:
```
1. 檢測網路中斷
   - Ping 失敗
   - API 請求超時
   ↓
2. 切換到離線模式
   - 使用本地快取資料
   - 交易記錄暫存本地佇列
   - 顯示離線狀態
   ↓
3. 離線期間:
   - 繼續營運 (使用本地資料)
   - 累積未上傳交易
   - 記錄離線時長
   ↓
4. 網路恢復:
   - 自動檢測連線
   - 觸發同步流程
   - 上傳累積交易
   - 下載更新資料
```

### 2. 同步佇列堆積

**情境**: 未上傳交易過多，佇列堆積

**處理流程**:
```
1. 檢測佇列大小
   - 佇列項目數 > 閾值 (1000 筆)
   ↓
2. 觸發警告
   - 通知管理員
   - 顯示佇列狀態
   ↓
3. 優化上傳:
   - 增加批次大小
   - 提高上傳頻率
   - 優先上傳重要交易
   ↓
4. 如持續堆積:
   - 檢查網路頻寬
   - 檢查中央處理能力
   - 考慮分散上傳
```

### 3. 重複交易檢測

**情境**: 同一交易被重複上傳

**處理流程**:
```
1. 中央接收交易
   ↓
2. 檢查交易 ID 是否已存在
   ↓
3. 如已存在:
   - 比對交易內容
   - 內容相同：忽略重複，回傳成功
   - 內容不同：標記為衝突，通知審核
   ↓
4. 本地收到回應:
   - 成功：從佇列移除
   - 衝突：標記待處理
```

## 🔌 API 調用

### 同步相關 API (依據 Config Sync Protocol 1.0.0)

| 順序 | API | 方法 | 說明 |
|------|-----|------|------|
| 1 | `/api/v1/machines/{id}/config/status` | GET | 查詢/檢查配置同步狀態 |
| 2 | `/api/v1/sync/upload` | POST | 上傳交易紀錄與日誌 |
| 3 | `/api/v1/configs/{id}` | GET | 下載特定配置內容 |
| 4 | `/api/v1/sync/download` | POST | 完整資料拉取 (含遊戲、設定) |
| 5 | `/api/v1/machines/{id}/config/status` | POST | 回報配置套用結果 |

## 💾 資料庫操作

### 涉及的資料表

| 資料表 | 操作 | 說明 |
|--------|------|------|
| `sync_queue` | INSERT/DELETE | 同步佇列 |
| `sync_logs` | INSERT | 同步日誌 |
| `sync_conflicts` | INSERT/UPDATE | 衝突記錄 |
| `transactions` | INSERT/SELECT | 交易紀錄 |

### 資料表結構

#### sync_queue

```sql
CREATE TABLE sync_queue (
    id VARCHAR(50) PRIMARY KEY,
    direction VARCHAR(10) NOT NULL, -- 'UP', 'DOWN'
    data_type VARCHAR(20) NOT NULL, -- 'TRANSACTION', 'CONFIG', 'GAME'
    data JSON NOT NULL,
    status VARCHAR(20) NOT NULL, -- 'PENDING', 'PROCESSING', 'COMPLETED', 'FAILED'
    retry_count INT DEFAULT 0,
    max_retries INT DEFAULT 5,
    next_retry_at TIMESTAMP,
    error_message TEXT,
    created_at TIMESTAMP NOT NULL,
    completed_at TIMESTAMP
);
```

#### sync_conflicts

```sql
CREATE TABLE sync_conflicts (
    id VARCHAR(50) PRIMARY KEY,
    data_type VARCHAR(20) NOT NULL,
    data_id VARCHAR(100) NOT NULL,
    local_version VARCHAR(50),
    central_version VARCHAR(50),
    local_data JSON,
    central_data JSON,
    resolution_strategy VARCHAR(20), -- 'CENTRAL_WINS', 'LOCAL_WINS', 'MANUAL'
    resolution_result VARCHAR(20), -- 'PENDING', 'RESOLVED'
    resolved_by VARCHAR(50),
    resolved_at TIMESTAMP,
    created_at TIMESTAMP NOT NULL
);
```

## 📊 同步參數

### 重試機制參數

| 參數 | 預設值 | 說明 |
|------|--------|------|
| 最大重試次數 | 5 次 | 失敗後最大重試次數 |
| 初始間隔 | 10 秒 | 第一次重試間隔 |
| 退避係數 | 2 | 指數退避係數 |
| 最大間隔 | 300 秒 | 最大重試間隔 (5 分鐘) |

### 佇列參數

| 參數 | 預設值 | 說明 |
|------|--------|------|
| 佇列警告閾值 | 1000 筆 | 超過此數量觸發警告 |
| 批次上傳大小 | 100 筆 | 每批次上傳交易數 |
| 上傳間隔 | 30 秒 | 定時上傳間隔 |

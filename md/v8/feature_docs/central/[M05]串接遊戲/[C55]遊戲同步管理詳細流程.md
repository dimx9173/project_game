# [C55] 遊戲同步管理詳細流程

**功能代碼**: [C55]  
**所屬模組**: [M05]串接遊戲  
**最後更新**: 2026-03-07  
**優先級**: 🔴 高  
**最後更新**: 2026-03-07 (v1.1 - 增加 API 金鑰加密儲存說明)

遊戲同步管理是集中式後台與第三方遊戲商 API 對接的核心功能，負責遊戲列表同步、遊戲狀態同步、以及異常處理。

## 🔄 完整流程圖

### 遊戲商 API 對接流程

```
┌─────────────────────────────────────────────────────────────┐
│  1. 管理員新增遊戲商                                        │
│     - 填寫遊戲商資訊 (名稱、API 端點、金鑰)                   │
│     - 選擇遊戲商類型 (電子、棋牌、捕魚、彩票)                │
│     ↓                                                       │
│  2. 系統測試連線                                            │
│     - 發送測試請求到遊戲商 API                              │
│     - 驗證金鑰有效性                                        │
│     - 記錄連線狀態                                          │
│     ↓                                                       │
│  3. 建立遊戲商記錄                                          │
│     - 記錄遊戲商資訊到資料庫                                │
│     - 狀態：待同步                                          │
│     ↓                                                       │
│  4. 觸發遊戲同步                                            │
│     - 手動觸發 / 定時觸發                                   │
│     - POST /api/providers/{id}/sync                        │
│     ↓                                                       │
│  5. 呼叫遊戲商 API 獲取遊戲列表                              │
│     - GET {provider_api}/games                              │
│     - 驗證回應格式                                          │
│     ↓                                                       │
│  6. 解析遊戲資料                                            │
│     - 遊戲 ID、名稱、類型                                    │
│     - RTP 率、最小/最大押注                                   │
│     - 遊戲圖示、預覽圖                                       │
│     - 支援語言、地區限制                                     │
│     ↓                                                       │
│  7. 比對現有遊戲                                            │
│     - 新增遊戲：資料庫中不存在                              │
│     - 更新遊戲：資訊有變更                                  │
│     - 下架遊戲：遊戲商已移除                                │
│     ↓                                                       │
│  8. 更新本地資料庫                                          │
│     - INSERT/UPDATE games 表                                 │
│     - 記錄同步時間                                          │
│     ↓                                                       │
│  9. 回報同步結果                                            │
│     - 新增數量、更新數量、下架數量                          │
│     - 錯誤訊息 (如有)                                       │
│     ↓                                                       │
│  10. 同步完成                                               │
│      - 更新遊戲商狀態：已同步                               │
│      - 記錄同步日誌                                          │
└─────────────────────────────────────────────────────────────┘
```

### 遊戲狀態同步流程

```
┌─────────────────────────────────────────────────────────────┐
│  定時檢查遊戲狀態 (每 5 分鐘)                                 │
│     ↓                                                       │
│  呼叫遊戲商 API 獲取遊戲狀態                                 │
│  GET {provider_api}/games/status                            │
│     ↓                                                       │
│  解析狀態資訊                                               │
│  - 遊戲 ID                                                   │
│  - 狀態：online, offline, maintenance                       │
│  - 維護訊息 (如適用)                                        │
│     ↓                                                       │
│  比對本地狀態                                               │
│  - 狀態變更：更新本地記錄                                   │
│  - 狀態相同：跳過                                           │
│     ↓                                                       │
│  更新本地資料庫                                             │
│  UPDATE games SET status = ?, maintenance_message = ?       │
│  WHERE game_id = ?                                          │
│     ↓                                                       │
│  通知前端狀態變更                                           │
│  - WebSocket 推播                                             │
│  - 前端更新遊戲列表顯示                                      │
└─────────────────────────────────────────────────────────────┘
```

## ⚠️ 異常處理

### 1. API 連線失敗

**情境**: 無法連線到遊戲商 API

**處理流程**:
```
1. 發送 API 請求超時 (30 秒)
   ↓
2. 重試機制:
   - 第 1 次重試：間隔 10 秒
   - 第 2 次重試：間隔 30 秒
   - 第 3 次重試：間隔 60 秒
   ↓
3. 3 次重試均失敗:
   - 標記遊戲商狀態：離線
   - 記錄錯誤事件
   - 通知管理員
   - 暫停該遊戲商的同步任務
```

### 2. API 回應格式錯誤

**情境**: 遊戲商 API 回應格式不符合預期

**處理流程**:
```
1. 解析 JSON 失敗
   ↓
2. 記錄原始回應
   ↓
3. 驗證回應格式:
   - 檢查必要欄位是否存在
   - 檢查資料類型是否正確
   ↓
4. 格式錯誤:
   - 記錄驗證失敗事件
   - 跳過本次同步
   - 通知管理員聯繫遊戲商
```

### 3. 遊戲資料不一致

**情境**: 同步的遊戲資料與本地記錄衝突

**處理流程**:
```
1. 比對遊戲資料
   ↓
2. 發現衝突 (如 RTP 率差異 > 5%):
   - 記錄衝突詳情
   - 標記為「待審核」
   - 不自動更新
   ↓
3. 通知管理員:
   - 顯示衝突詳情
   - 提供手動審核介面
   ↓
4. 管理員審核後:
   - 選擇接受新版本或保留舊版本
   - 記錄審核結果
```

### 4. 同步部分失敗

**情境**: 部分遊戲同步成功，部分失敗

**處理流程**:
```
1. 逐個處理遊戲資料
   ↓
2. 單個遊戲同步失敗:
   - 記錄失敗原因
   - 繼續處理下一個遊戲
   ↓
3. 同步完成後:
   - 回報成功數量、失敗數量
   - 提供失敗清單
   ↓
4. 管理員可選擇:
   - 重試失敗項目
   - 查看失敗原因
   - 手動修正
```

## 🔐 API 金鑰安全管理

### 加密儲存原則

**問題**: API 金鑰以明文儲存存在安全風險

**解決方案**: 使用 AES-256-GCM 加密儲存

### 加密架構

```
┌─────────────────────────────────────────────────────────────┐
│  金鑰管理架構                                                │
│                                                             │
│  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐   │
│  │   資料庫    │ ←── │  金鑰解密   │ ←── │   KMS/HSM   │   │
│  │ (加密儲存)  │     │   服務      │     │ (主金鑰)    │   │
│  └─────────────┘     └─────────────┘     └─────────────┘   │
│                              ↓                             │
│                      ┌─────────────┐                       │
│                      │  API 請求   │                       │
│                      │ (動態解密)  │                       │
│                      └─────────────┘                       │
└─────────────────────────────────────────────────────────────┘
```

### 加密流程

```
1. 管理員輸入 API 金鑰 (明文)
   ↓
2. 系統產生隨機 IV (Initialization Vector)
   ↓
3. 使用 KMS/HSM 中的主金鑰 (Master Key) 加密:
   - AES-256-GCM 加密
   - 產生認證標籤 (Auth Tag)
   ↓
4. 儲存加密結果:
   - api_key_encrypted = Base64(IV + CipherText + AuthTag)
   - api_key_version = 1 (每次更換金鑰遞增)
   ↓
5. 顯示成功訊息 (不顯示明文金鑰)
```

### 使用金鑰流程

```
1. 需要呼叫遊戲商 API
   ↓
2. 從資料庫讀取加密金鑰
   ↓
3. 解密金鑰 (使用 KMS):
   - 解析 Base64
   - 提取 IV, CipherText, AuthTag
   - 使用主金鑰解密
   - 驗證 AuthTag
   ↓
4. 將解密的金鑰加入請求 Header
   ↓
5. 執行 API 請求
   ↓
6. 請求完成後立即清除記憶體中的明文金鑰
```

### 金鑰輪換

```
1. 管理員更新 API 金鑰
   ↓
2. 系統加密新金鑰
   ↓
3. 更新資料庫:
   UPDATE game_providers 
   SET api_key_encrypted = @newEncryptedKey,
       api_key_version = api_key_version + 1,
       updated_at = NOW()
   WHERE id = @providerId;
   ↓
4. 記錄金鑰變更日誌 (不記錄金鑰內容)
   ↓
5. 驗證新金鑰可用性
   ↓
6. 完成輪換
```

### 環境變數配置

**嚴禁**:
- ❌ 將金鑰寫入程式碼
- ❌ 將金鑰寫入設定檔 (提交到 Git)
- ❌ 將金鑰放在未加密的資料庫欄位

**必須**:
- ✅ 使用環境變數
- ✅ 使用密碼管理工具 (1Password, Vault 等)
- ✅ 使用 KMS/HSM 服務

```bash
# 環境變數範例
export PROVIDER_API_KEY_MASTER="master-key-from-kms"
export DB_ENCRYPTION_KEY="encryption-key-from-kms"

# 或使用 Kubernetes Secret
kubectl create secret generic provider-keys \
  --from-literal=master-key="$MASTER_KEY"
```

### 資料庫欄位說明

| 欄位 | 類型 | 說明 |
|------|------|------|
| `api_key_encrypted` | VARCHAR(500) | 加密後的 API 金鑰 (Base64 編碼) |
| `api_key_version` | INT | 金鑰版本號，用於輪換追蹤 |

### SQL 範例 (加密/解密)

```sql
-- 插入加密的金鑰 (應用程式層處理加密)
INSERT INTO game_providers (id, name, api_endpoint, api_key_encrypted, provider_type, status, created_at, updated_at)
VALUES ('PROV001', '遊戲商A', 'https://api.provider-a.com', 
        'gAAAAABk...',  -- 加密後的金鑰
        'slot', 'ACTIVE', NOW(), NOW());

-- 查詢時不解密，僅返回標記
SELECT id, name, api_key_version, 
       '***ENCRYPTED***' as api_key  -- 不返回實際金鑰
FROM game_providers;
```

#### games

```sql
CREATE TABLE games (
    id VARCHAR(50) PRIMARY KEY,
    provider_id VARCHAR(50) NOT NULL,
    game_id VARCHAR(100) NOT NULL,
    name VARCHAR(200) NOT NULL,
    game_type VARCHAR(50) NOT NULL,
    rtp DECIMAL(5,2),
    min_bet DECIMAL(10,2),
    max_bet DECIMAL(10,2),
    status VARCHAR(20) NOT NULL, -- 'ONLINE', 'OFFLINE', 'MAINTENANCE'
    maintenance_message TEXT,
    icon_url VARCHAR(500),
    preview_url VARCHAR(500),
    languages JSON,
    regions JSON,
    is_enabled BOOLEAN DEFAULT TRUE,
    last_sync_at TIMESTAMP,
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL,
    FOREIGN KEY (provider_id) REFERENCES game_providers(id)
);
```

#### game_sync_logs

```sql
CREATE TABLE game_sync_logs (
    id VARCHAR(50) PRIMARY KEY,
    provider_id VARCHAR(50) NOT NULL,
    sync_type VARCHAR(20) NOT NULL, -- 'FULL', 'INCREMENTAL', 'STATUS'
    games_added INT DEFAULT 0,
    games_updated INT DEFAULT 0,
    games_removed INT DEFAULT 0,
    games_failed INT DEFAULT 0,
    error_message TEXT,
    started_at TIMESTAMP NOT NULL,
    completed_at TIMESTAMP,
    FOREIGN KEY (provider_id) REFERENCES game_providers(id)
);
```

## 📊 同步設定

### 定時同步設定

| 同步類型 | 頻率 | 說明 |
|---------|------|------|
| 完整同步 | 每日 02:00 | 同步所有遊戲資訊 |
| 狀態同步 | 每 5 分鐘 | 同步遊戲狀態 |
| 手動同步 | 隨時 | 管理員手動觸發 |

### 同步參數

| 參數 | 預設值 | 說明 |
|------|--------|------|
| API 超時 | 30 秒 | API 請求超時時間 |
| 重試次數 | 3 次 | 失敗重試次數 |
| 重試間隔 | 10/30/60 秒 | 遞增重試間隔 |
| 批次大小 | 100 | 每批次處理遊戲數 |

## ⚠️ 異常處理

### 1. 餘額不足

**情境**: 轉帳時來源餘額不足

**處理流程**:
```
1. 系統檢查來源餘額
   ↓
2. 餘額 < 轉帳金額
   ↓
3. 拒絕交易
   - 顯示錯誤訊息：「餘額不足，目前餘額：$XXX」
   - 建議操作：先充值或減少轉帳金額
   ↓
4. 記錄嘗試日誌
```

### 2. API 連線失敗

**情境**: 無法連線到遊戲商 API

**處理流程**:
```
1. 發送 API 請求超時 (30 秒)
   ↓
2. 重試機制:
   - 第 1 次重試：間隔 10 秒
   - 第 2 次重試：間隔 30 秒
   - 第 3 次重試：間隔 60 秒
   ↓
3. 3 次重試均失敗:
   - 標記交易失敗
   - 記錄錯誤事件
   - 通知管理員
   - 建議稍後重試
```

### 3. 限額超過

**情境**: 轉帳金額超過限額

**處理流程**:
```
1. 系統檢查限額
   ↓
2. 超過單筆限額:
   - 拒絕交易
   - 顯示限額資訊
   - 建議分批轉帳
   ↓
3. 超過單日/單月限額:
   - 拒絕交易
   - 顯示已使用限額
   - 建議明日/下月再執行
   - 或申請提高限額 (需審核)
```

### 4. 交易狀態不一致

**情境**: 遊戲商已扣款但本平台未收到確認

**處理流程**:
```
1. 發送轉帳請求後未收到回應
   ↓
2. 查詢交易狀態:
   - GET {provider_api}/transactions/{id}
   ↓
3. 比對結果:
   - 遊戲商已處理：同步狀態，完成交易
   - 遊戲商未處理：重新發送或取消
   ↓
4. 記錄不一致事件
   ↓
5. 必要時人工介入對帳
```

## 🔒 交易安全性

### 冪等性設計

**問題**: 網路重試可能導致重複轉帳

**解決方案**: 每次轉帳必須包含唯一金鑰，伺服器根據金鑰判斷是否已處理

#### 冪等性金鑰

```http
POST /api/wallet/transfer
Content-Type: application/json
Authorization: Bearer {token}

{
  "providerId": "PROV001",
  "amount": 10000,
  "direction": "IN",
  "remark": "週結歸集",
  "clientIdempotencyKey": "cli-20260307-001-a1b2c3d4"  // 客戶端產生
}

Response:
{
  "success": true,
  "transactionId": "TXN-20260307-001",
  "serverIdempotencyKey": "srv-20260307-001-e5f6g7h8"  // 伺服器產生
}
```

#### 處理流程

```
1. 客戶端發起轉帳請求
   - 產生 client_idempotency_key (UUID)
   - 包含在請求中
   ↓
2. 伺服器接收請求
   ↓
3. 檢查 client_idempotency_key 是否存在
   ↓
4. 根據不同狀態處理:
   
   a) 不存在 (新交易):
      - 建立新交易記錄
      - 狀態：PROCESSING
      - 執行轉帳
      - 完成後狀態：COMPLETED
      
   b) 已存在 + COMPLETED:
      - 這是重複請求
      - 直接返回現有交易結果
      - 不執行任何操作
      
   c) 已存在 + PROCESSING:
      - 這是重複請求 (客戶端等待中)
      - 返回處理中狀態
      - 客戶端輪詢等待完成
      
   d) 已存在 + FAILED:
      - 這是重試請求
      - 可以重新執行轉帳
      - 或建立新交易
```

#### 資料表結構 (新增欄位)

```sql
ALTER TABLE wallet_transactions ADD client_idempotency_key VARCHAR(100) UNIQUE;
ALTER TABLE wallet_transactions ADD server_idempotency_key VARCHAR(100) UNIQUE;
ALTER TABLE wallet_transactions ADD status VARCHAR(20) DEFAULT 'PROCESSING';  -- PENDING, PROCESSING, COMPLETED, FAILED
```

#### 冪等性檢查 SQL

```sql
-- 檢查是否存在
SELECT id, status, amount, balance_after 
FROM wallet_transactions 
WHERE client_idempotency_key = @clientKey;

-- 如果存在且已完成，直接返回
IF @@ROWCOUNT > 0 AND @existingStatus = 'COMPLETED'
BEGIN
    RETURN @existingTransaction;
END

-- 如果存在且處理中，等待
IF @existingStatus = 'PROCESSING'
BEGIN
    -- 客戶端輪詢等待
    WHILE @status = 'PROCESSING'
    BEGIN
        WAITFOR DELAY '00:00:01';
        SELECT @status = status FROM wallet_transactions WHERE client_idempotency_key = @clientKey;
    END
    RETURN @transaction;
END
```

### 交易鎖機制

**問題**: 併發操作可能導致餘額不一致

**解決方案**: 使用資料庫鎖或樂觀鎖確保併發安全

#### 方案一：樂觀鎖 (Optimistic Lock)

```sql
-- 使用 version 欄位實現樂觀鎖
ALTER TABLE provider_wallets ADD version INT DEFAULT 1 NOT NULL;

-- 餘額更新時檢查版本
UPDATE provider_wallets 
SET balance = balance + @amount,
    version = version + 1,
    updated_at = NOW()
WHERE provider_id = @providerId
  AND version = @expectedVersion  -- 只有版本匹配才更新
  AND balance + @amount >= 0;  -- 餘額不能為負

-- 檢查更新結果
IF @@ROWCOUNT = 0
BEGIN
    -- 版本衝突或餘額不足
    SELECT error = CASE 
        WHEN NOT EXISTS (SELECT 1 FROM provider_wallets WHERE provider_id = @providerId)
        THEN '錢包不存在'
        WHEN (SELECT balance FROM provider_wallets WHERE provider_id = @providerId) + @amount < 0
        THEN '餘額不足'
        ELSE '版本衝突，請重試'
    END;
END
```

#### 方案二：悲觀鎖 (Pessimistic Lock)

```sql
-- 使用 SELECT FOR UPDATE 鎖定記錄
BEGIN TRANSACTION;

-- 鎖定錢包記錄
SELECT balance, version 
FROM provider_wallets 
WITH (UPDLOCK, ROWLOCK)
WHERE provider_id = @providerId;

-- 檢查餘額
IF @balance + @amount < 0
BEGIN
    ROLLBACK;
    RAISERROR('餘額不足', 16, 1);
END

-- 更新餘額
UPDATE provider_wallets 
SET balance = balance + @amount,
    updated_at = NOW()
WHERE provider_id = @providerId;

COMMIT;
```

#### 方案三：餘額檢查 (atomic update)

```sql
-- 原子操作：餘額不足時不更新
UPDATE provider_wallets 
SET balance = balance + @amount,
    updated_at = NOW()
WHERE provider_id = @providerId
  AND balance + @amount >= 0;  -- 餘額檢查在 WHERE 條件中

-- 檢查是否更新成功
IF @@ROWCOUNT = 0
BEGIN
    RAISERROR('餘額不足', 16, 1);
END
```

### 完整轉帳交易 (冪等性 + 鎖)

```sql
CREATE PROCEDURE sp_WalletTransfer
    @providerId VARCHAR(50),
    @amount DECIMAL(12,2),
    @direction VARCHAR(10),
    @clientIdempotencyKey VARCHAR(100),
    @operatorId VARCHAR(50),
    @remark TEXT
AS
BEGIN
    SET NOCOUNT ON;
    
    DECLARE @transactionId VARCHAR(50);
    DECLARE @existingStatus VARCHAR(20);
    DECLARE @balance DECIMAL(12,2);
    
    -- 1. 檢查冪等性
    SELECT @transactionId = id, @existingStatus = status 
    FROM wallet_transactions 
    WHERE client_idempotency_key = @clientIdempotencyKey;
    
    IF @transactionId IS NOT NULL
    BEGIN
        -- 已存在，根據狀態返回
        IF @existingStatus = 'COMPLETED'
        BEGIN
            SELECT * FROM wallet_transactions WHERE id = @transactionId;
            RETURN 0;
        END
        
        IF @existingStatus = 'PROCESSING'
        BEGIN
            -- 等待處理完成
            WHILE EXISTS (SELECT 1 FROM wallet_transactions 
                         WHERE id = @transactionId AND status = 'PROCESSING')
            BEGIN
                WAITFOR DELAY '00:00:01';
            END
            
            SELECT * FROM wallet_transactions WHERE id = @transactionId;
            RETURN 0;
        END
    END
    
    -- 2. 建立新交易
    SET @transactionId = 'TXN-' + CONVERT(VARCHAR, GETDATE(), 112) + '-' + RIGHT('0000' + CAST(NEWID() % 10000 AS VARCHAR), 4);
    
    BEGIN TRANSACTION;
    
    BEGIN TRY
        -- 3. 鎖定錢包記錄
        SELECT @balance = balance 
        FROM provider_wallets 
        WITH (UPDLOCK, ROWLOCK)
        WHERE provider_id = @providerId;
        
        -- 4. 檢查餘額
        IF @balance + @amount < 0
        BEGIN
            ROLLBACK;
            RAISERROR('餘額不足', 16, 1);
        END
        
        -- 5. 插入交易記錄
        INSERT INTO wallet_transactions (id, provider_id, transaction_type, amount, 
            balance_before, balance_after, status, client_idempotency_key, 
            server_idempotency_key, remark, operator_id, created_at)
        VALUES (@transactionId, @providerId, 
            CASE WHEN @direction = 'IN' THEN 'TRANSFER_IN' ELSE 'TRANSFER_OUT' END,
            @amount, @balance, @balance + @amount, 'PROCESSING',
            @clientIdempotencyKey, 'srv-' + @transactionId, 
            @remark, @operatorId, GETDATE());
        
        -- 6. 更新餘額
        UPDATE provider_wallets 
        SET balance = balance + @amount,
            updated_at = GETDATE()
        WHERE provider_id = @providerId;
        
        -- 7. 更新交易狀態
        UPDATE wallet_transactions 
        SET status = 'COMPLETED', 
            completed_at = GETDATE()
        WHERE id = @transactionId;
        
        COMMIT;
        
        -- 8. 返回結果
        SELECT * FROM wallet_transactions WHERE id = @transactionId;
        
    END TRY
    BEGIN CATCH
        -- 發生錯誤，回滾
        ROLLBACK;
        
        -- 更新交易狀態為失敗
        IF @transactionId IS NOT NULL
        BEGIN
            UPDATE wallet_transactions 
            SET status = 'FAILED', 
                completed_at = GETDATE(),
                remark = ERROR_MESSAGE()
            WHERE id = @transactionId;
        END
        
        THROW;
    END CATCH
END
```

### 涉及的資料表

| 資料表 | 操作 | 說明 |
|--------|------|------|
| `provider_wallets` | INSERT/UPDATE | 記錄遊戲商錢包資訊 |
| `wallet_transactions` | INSERT | 記錄轉帳交易 |
| `wallet_limits` | INSERT/UPDATE | 記錄限額設定 |
| `auto_collect_settings` | INSERT/UPDATE | 記錄歸集設定 |
| `operation_logs` | INSERT | 記錄操作日誌 |

### 資料表結構

#### provider_wallets

```sql
CREATE TABLE provider_wallets (
    provider_id VARCHAR(50) PRIMARY KEY,
    balance DECIMAL(12,2) NOT NULL DEFAULT 0,
    frozen_balance DECIMAL(12,2) NOT NULL DEFAULT 0,
    currency VARCHAR(10) DEFAULT 'TWD',
    last_sync_at TIMESTAMP,
    status VARCHAR(20) NOT NULL, -- 'ACTIVE', 'INACTIVE', 'SUSPENDED'
    updated_at TIMESTAMP NOT NULL
);
```

#### wallet_transactions

```sql
CREATE TABLE wallet_transactions (
    id VARCHAR(50) PRIMARY KEY,
    provider_id VARCHAR(50) NOT NULL,
    transaction_type VARCHAR(20) NOT NULL, -- 'TRANSFER_IN', 'TRANSFER_OUT'
    amount DECIMAL(12,2) NOT NULL,
    balance_before DECIMAL(12,2),
    balance_after DECIMAL(12,2),
    status VARCHAR(20) NOT NULL, -- 'PENDING', 'COMPLETED', 'FAILED'
    provider_transaction_id VARCHAR(100),
    remark TEXT,
    operator_id VARCHAR(50) NOT NULL,
    created_at TIMESTAMP NOT NULL,
    completed_at TIMESTAMP,
    FOREIGN KEY (provider_id) REFERENCES provider_wallets(provider_id)
);
```

#### wallet_limits

```sql
CREATE TABLE wallet_limits (
    id VARCHAR(50) PRIMARY KEY,
    provider_id VARCHAR(50) NOT NULL,
    single_min DECIMAL(12,2) DEFAULT 100,
    single_max DECIMAL(12,2) DEFAULT 100000,
    daily_max DECIMAL(12,2) DEFAULT 500000,
    monthly_max DECIMAL(12,2) DEFAULT 10000000,
    updated_at TIMESTAMP NOT NULL,
    updated_by VARCHAR(50) NOT NULL,
    FOREIGN KEY (provider_id) REFERENCES provider_wallets(provider_id)
);
```

### 限額調整流程

```
1. 管理員申請調整限額
   ↓
2. 填寫申請資訊:
   - 申請調整的限額類型
   - 目前限額
   - 申請新限額
   - 調整原因
   ↓
3. 系統驗證申請:
   - 檢查申請人權限
   - 檢查調整幅度 (超過 50% 需高階審核)
   ↓
4. 審核流程:
   - 一般調整：主管審核
   - 大幅調整：高階主管審核
   ↓
5. 審核通過:
   - 更新限額設定
   - 記錄變更日誌
   - 通知申請人
   ↓
6. 審核拒絕:
   - 記錄拒絕原因
   - 通知申請人
```

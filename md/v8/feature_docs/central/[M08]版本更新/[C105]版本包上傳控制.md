**最後更新**: 2026-03-06

## 功能說明

### 上傳新版本與版本包控制

**操作流程**：

```
1. 點擊「上傳版本」按鈕
2. 選擇版本檔案
3. 填寫版本資訊
4. 系統計算 SHA-256
5. 驗證檔案完整性
6. 完成上傳
```

**輸入欄位**：

| 欄位名稱 | 類型 | 必填 | 說明         |
| -------- | ---- | ---- | ------------ |
| 版本號碼 | 文字 | ✅   | 格式：v8.0.x |
| 版本說明 | 文字 | ✅   | 更新內容描述 |
| 檔案     | 檔案 | ✅   | .zip 格式    |
| 目標機型 | 多選 | ✅   | 適用機型     |
| 強制更新 | 開關 | -    | 是否強制更新 |

**檔案驗證**：

- 自動計算 MD5
- 自動計算 SHA-256
- 驗證檔案大小
- 檢查壓縮完整性

## OTA 遠端更新 (Over-The-Air)

### 版本控制

- **中央版控**：統一管理所有本地後台與附屬設備的版本。
- **版本號規則**：遵循 Semantic Versioning (Major.Minor.Patch)。
- **版本相容性矩陣**：定義不同硬體型號與軟體版本的相容性。

### 部署策略

1. **灰度測試 (Canary Release)**
   - 選擇特定機台或店鋪先行更新 (如 5%)
   - 監控 24 小時後無異常，再全面推送
2. **定時更新 (Scheduled Update)**
   - 設定於離峰時段 (如凌晨 03:00 - 05:00) 自動下載並安裝
3. **強制更新 (Force Update)**
   - 若遇重大安全漏洞或核心 Bug，執行強制中斷遊戲並更新

### 斷點續傳機制

- 機台下載更新包時，若遇網路中斷，可從中斷的 Byte 繼續下載。
- 下載完成後，強制核對 **SHA-256 Checksum**，確保檔案未損毀。

### 更新失敗回滾 (Rollback)

- 更新前自動備份當前系統核心檔案。
- 若安裝失敗或啟動後核心進程 (如通信服務) 異常，自動回滾至上一穩定版本，並向中央發送 `UpdateFailed` 事件。

## [C11] 開分/洗分詳細流程

### ✅ 優點

1. **流程完整**: 從登入到完成交易的每個步驟都有詳細說明
2. **異常處理豐富**: 涵蓋餘額不足、PIN驗證失敗、連線中斷等場景
3. **風控規則明確**: 單筆限額、單日累計限額、異常交易偵測都有具體數值
4. **API 設計合理**: 端點命名清晰，請求/回應格式完整

### ⚠️ 問題

#### 1. 交易原子性問題 (🔴 高)

**問題描述**: 
```sql
-- 文件中的交易範例缺少錯誤處理和 ROLLBACK
BEGIN TRANSACTION;
INSERT INTO cash_orders ...
UPDATE machine_balance ...
INSERT INTO operation_logs ...
UPDATE cash_orders ...
COMMIT;
```

**風險**: 如果中間任何一步失敗，沒有明確的 ROLLBACK 機制

**建議**: 
```sql
BEGIN TRANSACTION;
BEGIN TRY
    INSERT INTO cash_orders ...
    UPDATE machine_balance ...
    INSERT INTO operation_logs ...
    UPDATE cash_orders ...
    COMMIT;
END TRY
BEGIN CATCH
    ROLLBACK;
    -- 記錄錯誤
    THROW;
END CATCH
```

#### 2. PIN 長度不一致 (🟡 中)

**問題描述**: 
- 文件開頭提到「輸入 8 位值班 PIN」
- 但在規格書中提到「PIN 碼必須為 4-6 位數字」

**風險**: 開發時可能產生混淆

**建議**: 統一 PIN 碼長度規範，建議為 6 位數字

#### 3. 缺少冪等性設計 (🟡 中)

**問題描述**: 
- 開分/洗分 API 沒有提到冪等性設計
- 如果客戶端重複發送相同請求，可能導致重複扣款/入帳

**建議**: 
```http
POST /api/machines/{id}/cash-in
X-Idempotency-Key: uuid-v4

// Server 端檢查相同的 Idempotency-Key 是否已處理
```

#### 4. 超時重試機制不明確 (🟡 中)

**問題描述**: 
- 文件提到「3 次重試」，但沒有說明重試間隔策略
- 沒有提到指數退避

**建議**: 採用指數退避重試（與 L36 保持一致）

### 💡 建議

1. **加入分散式鎖**: 防止同一機台同時被多個操作員操作
2. **加入交易流水號**: 每筆交易產生唯一流水號，方便追蹤
3. **加入餘額快照**: 記錄交易前後餘額，方便對帳

## [C74] OTA 版本更新詳細流程

### ✅ 優點

1. **流程設計完整**: 從上傳、驗證、下載、安裝到驗證，步驟清晰
2. **斷點續傳支援**: 下載失敗時支援 Range Header 續傳
3. **自動回滾機制**: 安裝失敗時自動恢復舊版本
4. **版本號規範明確**: 使用 SemVer，比對邏輯清晰
5. **資料表設計合理**: `ota_task_machines` 可以追蹤每台機器的更新狀態

### ⚠️ 問題

#### 1. 缺少簽章驗證 (🟡 中)

**問題描述**: 
- 文件提到「檢查簽章」但沒有說明簽章演算法和流程
- 沒有說明如何產生和驗證簽章

**建議**: 
```markdown
### 版本包簽章流程

1. 上傳時計算 SHA256 雜湊值
2. 使用 RSA-2048 私鑰簽章
3. 機台下載後使用公鑰驗證簽章
4. 驗證通過才執行安裝
```

#### 2. 下載進度追蹤不完整 (🟢 低)

**問題描述**: 
- `ota_task_machines` 有 `download_progress` 欄位
- 但沒有說明進度更新頻率和機制

**建議**: 每下載 10% 或每隔 5 秒回報一次進度

#### 3. 安裝超時後的清理 (🟡 中)

**問題描述**: 
- 安裝超時 10 分鐘後終止腳本並回滾
- 但沒有說明如何處理「腳本卡住無法終止」的情況

**建議**: 加入強制重啟機制

### 💡 建議

1. **加入灰度發布**: 先選擇少量機台測試，再逐步擴大
2. **加入版本回退 API**: 允許手動回退到指定版本
3. **加入安裝前置檢查**: 檢查磁碟空間、記憶體是否足夠

## [C55] 遊戲同步管理詳細流程

### ✅ 優點

1. **同步類型分類清楚**: 完整同步、狀態同步、手動同步
2. **異常處理完善**: API 連線失敗、格式錯誤、資料衝突都有處理
3. **定時同步設定明確**: 每日 02:00 完整同步，每 5 分鐘狀態同步
4. **資料表設計合理**: `game_sync_logs` 可以追蹤每次同步結果

### ⚠️ 問題

#### 1. 缺少同步鎖 (🟡 中)

**問題描述**: 
- 沒有提到防止同步任務重疊的機制
- 如果上一次同步還沒完成，新的同步開始可能導致資料不一致

**建議**: 
```sql
-- 加入同步鎖
UPDATE game_providers 
SET sync_status = 'SYNCING', 
    sync_locked_at = NOW()
WHERE id = ? AND sync_status != 'SYNCING';

-- 檢查是否獲得鎖
SELECT sync_locked_at FROM game_providers WHERE id = ?;
```

#### 2. 遊戲商 API 回應格式驗證不足 (🟡 中)

**問題描述**: 
- 文件提到「驗證回應格式」但沒有說明具體驗證規則
- 不同遊戲商可能有不同的 API 格式

**建議**: 
```javascript
// 定義標準回應格式 Schema
const gameSchema = {
  gameId: { type: 'string', required: true },
  name: { type: 'string', required: true },
  type: { type: 'string', enum: ['slot', 'table', 'fish', 'lottery'] },
  rtp: { type: 'number', min: 0, max: 100 },
  // ...
};

// 使用 Schema 驗證
validateGameResponse(response, gameSchema);
```

#### 3. 缺少 API 金鑰加密儲存說明 (🔴 高)

**問題描述**: 
- `game_providers` 表的 `api_key` 欄位沒有說明加密方式
- API 金鑰是敏感資訊，應該加密儲存

**建議**: 
```sql
api_key_encrypted VARCHAR(500) NOT NULL, -- 使用 AES-256-GCM 加密
api_key_iv VARCHAR(64) NOT NULL,         -- 加密 IV
```

#### 4. 同步失敗後的自動恢復 (🟢 低)

**問題描述**: 
- 同步失敗後只是「通知管理員」
- 沒有自動恢復機制

**建議**: 加入定時重試機制（例如每小時重試失敗的同步）

### 💡 建議

1. **加入遊戲商適配器模式**: 不同遊戲商使用不同的適配器處理 API 差異
2. **加入同步佇列**: 使用佇列處理同步任務，避免同步重疊
3. **加入 Webhook 接收**: 讓遊戲商主動推送更新，減少輪詢

## [L34-L37] 同步對帳詳細流程

### ✅ 優點

1. **雙向佇列設計優秀**: 上行/下行佇列分離，職責清晰
2. **指數退避重試完善**: 參數明確，最大重試次數、間隔都有定義
3. **衝突解決策略多元**: 中央優先、本地優先、人工審核三種策略
4. **離線模式支援**: 網路中斷時可繼續營運
5. **資料表設計完整**: `sync_queue` 和 `sync_conflicts` 可以追蹤所有同步狀態

### ⚠️ 問題

#### 1. 衝突檢測觸發條件不明確 (🟡 中)

**問題描述**: 
- 文件提到「比對版本號/時間戳」但沒有說明何時觸發衝突檢測
- 是每次同步都檢測？還是特定操作才檢測？

**建議**: 
```markdown
### 衝突檢測觸發時機

1. **下行同步**: 收到中央更新時，比對本地版本號
2. **上行同步**: 上傳交易前，檢查本地是否有未同步的修改
3. **定時對帳**: 每日 03:00 執行完整對帳，檢測不一致
```

#### 2. 佇列優先級設計缺失 (🟡 中)

**問題描述**: 
- 所有同步項目放在同一佇列
- 沒有優先級區分（例如：交易上傳應優先於操作日誌）

**建議**: 
```sql
ALTER TABLE sync_queue ADD COLUMN priority INT DEFAULT 0;
-- 0: 一般, 1: 重要, 2: 緊急
-- 交易上傳: priority = 2
-- 操作日誌: priority = 0
```

#### 3. 重複交易檢測邏輯可能漏判 (🟡 中)

**問題描述**: 
- 文件提到「檢查交易 ID 是否已存在」
- 但如果本地因網路問題重複產生相同交易但不同 ID，就無法檢測

**建議**: 加入「交易指紋」檢測
```javascript
// 基於交易關鍵資訊產生指紋
const fingerprint = hash(`${machineId}-${amount}-${timestamp}-${type}`);
// 檢查指紋是否重複
```

#### 4. 佇列堆積處理不夠積極 (🟢 低)

**問題描述**: 
- 佇列堆積時只是「增加批次大小、提高上傳頻率」
- 沒有考慮到中央處理能力

**建議**: 加入「智慧壓縮」機制，合併相同類型的交易

### 💡 建議

1. **加入同步狀態儀表板**: 視覺化佇列狀態、衝突數量
2. **加入自動衝突解決**: 根據資料類型自動選擇最佳策略
3. **加入同步效能監控**: 記錄同步耗時、成功率

## [C106] 代理錢包管理詳細流程

### ✅ 優點

1. **流程完整**: 餘額查詢、轉帳、歸集設定、限額控制都有詳細說明
2. **限額設計合理**: 單筆、單日、單月限額分層控制
3. **限額調整流程完善**: 需要審核，大幅調整需要高階審核
4. **資料表設計完整**: `wallet_transactions` 記錄了交易前後餘額

### ⚠️ 問題

#### 1. 缺少交易冪等性設計 (🔴 高)

**問題描述**: 
- 轉帳 API 沒有提到冪等性設計
- 如果網路超時重複發送，可能導致重複轉帳

**建議**: 
```http
POST /api/wallet/transfer
X-Idempotency-Key: uuid-v4

// Server 端:
// 1. 檢查 Idempotency-Key 是否已處理
// 2. 如已處理，返回之前的結果
// 3. 如未處理，執行轉帳並記錄 Key
```

#### 2. 缺少交易鎖 (🔴 高)

**問題描述**: 
- 餘額查詢和轉帳可能併發執行
- 可能導致「餘額檢查通過但轉帳時餘額已不足」

**建議**: 
```sql
-- 使用悲觀鎖
SELECT * FROM provider_wallets 
WHERE provider_id = ? 
FOR UPDATE;

-- 執行轉帳
UPDATE provider_wallets 
SET balance = balance - ? 
WHERE provider_id = ?;

COMMIT;
```

#### 3. 歸集執行時機不夠明確 (🟡 中)

**問題描述**: 
- 文件提到「設定歸集時間（定時執行）」
- 但沒有說明歸集任務如何排程

**建議**: 使用 Cron 表達式定義歸集時間
```markdown
### 歸集排程設定

- 每日歸集: `0 2 * * *` (每日 02:00)
- 每週歸集: `0 2 * * 0` (每週日 02:00)
- 每月歸集: `0 2 1 * *` (每月 1 日 02:00)
```

#### 4. 缺少遊戲商 API 超時處理 (🟡 中)

**問題描述**: 
- 文件提到「等待回應 (timeout: 30 秒)」
- 但沒有說明超時後如何查詢交易狀態

**建議**: 
```markdown
### 超時後的交易狀態查詢

1. 發送轉帳請求後 30 秒未收到回應
2. 查詢遊戲商交易狀態 API:
   GET {provider_api}/transactions/{transactionId}
3. 根據查詢結果更新本地狀態
4. 如仍無法確定，標記為「待人工審核」
```

### 💡 建議

1. **加入交易通知機制**: 轉帳完成後主動通知相關人員
2. **加入餘額預警**: 遊戲商餘額過低時自動提醒充值
3. **加入多幣別支援**: 如需支援多種貨幣，需加入匯率轉換

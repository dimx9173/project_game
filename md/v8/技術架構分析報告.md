## 2. 風險項目 (🔴 紅色)

### 🔴 R01: 心跳間隔不一致

**嚴重程度**: 高  
**影響範圍**: 機台管理、監控中心

| 文件 | 描述 |
|-----|------|
| 規格書 4.7.1 | 「心跳間隔：預設 **30 秒**」 |
| 規格書 5.6.2 | 「Heartbeat 間隔不得超過 **60 秒**」 |
| SD 文件 | 「機台**每分鐘**發送心跳」 |
| 功能矩陣 | 「機台**每分鐘**發送心跳」 |

**風險**: 
- 開發人員可能採用不同數值
- 監控告警邏輯可能不一致
- 離線判定時間計算錯誤

**建議**: 統一為「**預設 30 秒，可配置 10-300 秒，最大 60 秒**」

---

### 🔴 R02: 交易簽名演算法表述不清

**嚴重程度**: 高  
**影響範圍**: 安全性、交易驗證

| 文件 | 描述 |
|-----|------|
| 規格書 5.3 | 「交易簽名驗證（**HMAC-SHA256**）」 |
| 規格書 6.2 | 「所有交易使用 **RSA** 數位簽名進行驗證」 |
| SA 文件 6.3 | 「所有交易使用 **RSA** 數位簽名進行驗證」 |
| SD 文件 | 「**HMAC-SHA256** 用於資料完整性」、「**RSA-2048** 用於 API 簽名」 |

**風險**: 
- 開發人員可能實作錯誤的簽名驗證
- 安全漏洞
- 與遊戲商 API 整合失敗

**建議**: 明確區分使用場景
- **RSA-2048**: 用於 API 簽名、機台認證
- **HMAC-SHA256**: 用於資料完整性驗證

---

### 🔴 R03: Redis 快取未詳細說明

**嚴重程度**: 中  
**影響範圍**: 效能設計、部署架構

**問題**:
- 僅規格書提及 Redis 7+
- SA 文件、資料庫設計文件未提及 Redis 使用場景
- 未說明快取策略、過期時間、失效機制

**風險**: 
- 開發人員可能遺漏 Redis 相關實作
- 效能瓶頸
- 部署架構不完整

**建議**: 補充 Redis 使用場景說明
- Session 管理
- API 回應快取
- 即時數據緩存（儀表板）
- 分散式鎖（交易處理）

---

### 🔴 R04: 功能矩陣日期不同步

**影響範圍**: 文件一致性

## 3. 待討論項目 (🟡 黃色)

### 🟡 D01: WebSocket vs SignalR 表述

**問題**: 規格書寫「WebSocket」，SA 文件寫「SignalR」

**建議**: 統一表述為「**SignalR (基於 WebSocket)**」

---

### 🟡 D02: 遊戲測試功能架構說明

**現狀**: 
- 功能矩陣、資料庫設計、API 都有設計
- SA 文件未詳細說明遊戲測試架構

**建議**: SA 文件補充：
- 測試模式啟用流程
- 虛擬錢包與真實錢包隔離
- 測試報告生成機制
- 權限控制（僅系統管理員、維修外包）

---

### 🟡 D03: 多人遊戲即時性需求

**現狀**: 規格書規定「牌桌狀態每 5 秒更新一次」

**問題**: 
- 5 秒更新對多人遊戲是否足夠即時？
- 是否需要 SignalR 即時推送？

**建議**: 評估即時性需求，考慮：
- 牌桌狀態即時推送
- 下注結果即時回饋
- 牌局歷史延遲更新

---

### 🟡 D04: 本地端 PIN 碼安全設計

**現狀**: 
- 規格書規定「PIN 錯誤 5 次鎖定」
- 資料庫設計有 `pinHash` 欄位
- 未詳細說明 PIN 碼重置流程

**建議**: 補充：
- PIN 碼重置權限與流程
- 鎖定後解鎖機制
- PIN 碼加密儲存方式

---

### 🟡 D05: 爭議對帳處理流程細節

**現狀**: 
- 規格書有爭議類型、處理方式
- 未詳細說明「人工介入」的權限層級

**建議**: 確認：
- 爭議單建立權限
- 裁決權限層級
- 主管授權碼機制
- 大額爭議的審核流程

---

### 🟡 D06: OTA 更新失敗還原機制

**現狀**: 規格書規定「更新失敗自動還原至上一版本」

**問題**: 
- 如何判斷更新失敗？
- 還原觸發條件？
- 還原過程中的資料保護？

**建議**: 補充：
- 更新失敗檢測邏輯
- 還原點建立時機
- 還原過程的交易保護

│                    文件同步建議                              │

│  1. 建立「規格變更清單」文件，記錄所有變更                    │

### 4.3 API 規格補充

建議補充以下 API 端點說明：

| 端點 | 說明 |
|-----|------|

## 1. 文件概述

本文件詳細闡述遊戲平台管理系統的資料庫設計，包含集中式後台與單機本地後台兩個層次的資料庫架構、資料表設計、索引策略、備份策略等內容。

## 2. 資料庫設計原則

### 2.1 規範化設計

遵循資料庫規範化原則，消除資料冗餘，提高資料一致性。採用第三規範化形式（3NF）進行設計。

### 2.2 效能優化

根據查詢頻率和業務需求，適度進行反規範化設計，提高查詢效能。使用適當的索引策略加速查詢。

### 2.3 可擴展性

設計支援資料分片與分表，為未來的資料增長預留擴展空間。

### 2.4 安全性

敏感資料加密儲存，實施嚴格的存取控制，記錄詳細的稽核日誌。

### 2.5 本地端快取原則（中央集權架構）

> **核心設計**：本地端資料為雲端快取（唯讀），無法直接修改資料，確保資料一致性。

- **唯讀快取**：遊戲、玩家、系統設定等資料由雲端統一管理，本地僅做快取
- **寫入經雲端**：所有資料變更透過 API 送至雲端處理，本地不直接寫入資料表
- **本地佇列**：交易紀錄等需要上傳的資料，先存入本地佇列，連線時同步至雲端
- **離線受限**：未連線至雲端時，僅能查看快取資料，無法開始新遊戲

### 3.3 機台識別機制

機台必須向集中式註冊才能啟用，**由集中式產生 `instanceId` 並綁定 MAC**。

> **設計原則**：**必須連線才能玩**。機台離線後無法開始新遊戲。
> **本地端限制**：本地端資料為雲端快取，**無法直接修改**，所有變更須透過雲端 API。

| 欄位名稱 | 說明 |
| -------- | ---- |
| `instanceId` | 機台實例 ID（由集中式產生，綁定 MAC） |
| `mac` | 網卡 MAC 位址（註冊時綁定） |
| `registeredAt` | 首次向集中式註冊時間 |
| `lastSeen` | 最後上線時間 |
| `status` | 註冊狀態（registered / offline / suspended） |

> **說明**：
> - `instanceId` 由集中式產生，確保唯一性
> - 註冊時 MAC 與 `instanceId` 綁定
> - `machines` 表即為單機本地後台的實例表，用於追蹤資料來源與遊戲連線狀態

### 3.4 玩家 ID 生成規則

> **設計原則**：玩家可跨系統追蹤，ID 由集中式統一產生並綁定帳號，註冊時需連線至集中式。

```
id = 集中式統一產生（綁定帳號）
```

- 玩家註冊時需連線至集中式，由集中式統一分配 ID
- ID 跨系統唯一，確保跨系統追蹤
- **衝突處理**：碰撞玩家進入「爭議對帳處理專區」

> **說明**：管理第三方遊戲供應商（如：電子遊戲、棋牌、捕魚、彩票等遊戲商）。

| 欄位名稱       | 資料類型     | 約束條件        | 說明         |

| `id`           | VARCHAR(50)  | PRIMARY KEY     | 遊戲商編號   |
| `name`         | VARCHAR(100) | NOT NULL UNIQUE | 遊戲商名稱   |
| `code`         | VARCHAR(20)  | NOT NULL UNIQUE | 遊戲商代碼（用於 API 串接）|

| `heartbeatUrl` | VARCHAR(255) |                 | 心跳檢測 URL |

| `lastHeartbeat`| DATETIME     |                 | 最後心跳時間 |

#### 業務規則

| 規則編號 | 規則說明                    |
| -------- | --------------------------- |
| PRV-001  | 遊戲商 API Key 必須加密儲存（AES-256-GCM）|
| PRV-002  | 單一遊戲商最多 100 個遊戲   |
| PRV-003  | 對帳差異超過 1% 發出告警    |
| PRV-004  | 心跳間隔預設 30 秒，連續 3 次失敗標記為離線 |

## 6. 索引策略

## 7. 備份與恢復策略

### 7.1 集中式後台備份

每日進行完整備份，備份儲存在獨立的備份伺服器。每小時進行增量備份，支援時間點恢復。備份檔案加密儲存，定期驗證備份完整性。

### 7.2 單機本地後台備份

> **簡化設計**：本地端資料主要為雲端快取，備份需求簡化。

- **本地日誌備份**：本地交易佇列、日誌等資料手動/自動備份
- **雲端同步**：所有重要資料由雲端統一管理，本地僅作暫存
- **加密儲存**：本地備份檔案加密儲存

### 7.3 恢復流程

- **集中式後台**：由資料庫管理員進行恢復，需要驗證備份完整性，恢復前進行備份。
- **單機本地後台**：本地快取可直接從雲端重新同步，無需複雜的本地恢復流程。

## 8. 資料安全

### 8.1 加密

敏感欄位（密碼、交易簽名）使用 **AES-256-GCM** 加密演算法進行加密。資料庫連線使用 SSL/TLS 加密協議。

### 8.2 存取控制

實施嚴格的資料庫使用者權限管理，不同使用者擁有不同的操作權限。應用程式使用專用的資料庫帳號進行連線。

### 8.3 稽核日誌

記錄所有重要操作的詳細日誌，包括操作者、操作時間、操作內容、操作結果等資訊。定期審查稽核日誌，發現異常操作。

## 1. 文件概述

本文件詳細闡述遊戲平台管理系統的資料庫設計，包含集中式後台與單機本地後台兩個層次的資料庫架構、資料表設計、索引策略、備份策略等內容。

## 2. 資料庫設計原則

### 2.1 規範化設計

遵循資料庫規範化原則，消除資料冗餘，提高資料一致性。採用第三規範化形式（3NF）進行設計。

### 2.2 效能優化

根據查詢頻率和業務需求，適度進行反規範化設計，提高查詢效能。使用適當的索引策略加速查詢。

### 2.3 可擴展性

設計支援資料分片與分表，為未來的資料增長預留擴展空間。

### 2.4 安全性

敏感資料加密儲存，實施嚴格的存取控制，記錄詳細的稽核日誌。

### 2.5 本地端快取原則（中央集權架構）

> **核心設計**：本地端資料為雲端快取（唯讀），無法直接修改資料，確保資料一致性。

- **唯讀快取**：遊戲、機台設定、系統設定等資料由雲端統一管理，本地僅做快取
- **寫入經雲端**：所有資料變更透過 API 送至雲端處理，本地不直接寫入資料表
- **本地佇列**：交易紀錄等需要上傳的資料，先存入本地佇列，連線時同步至雲端
- **離線受限**：未連線至雲端時，僅能查看快取資料，無法開始新遊戲

### 3.3 機台識別機制

機台必須向集中式註冊才能啟用，**由集中式產生 `instanceId` 並綁定 MAC**。

> **設計原則**：**必須連線才能玩**。機台離線後無法開始新遊戲。
> **本地端限制**：本地端資料為雲端快取，**無法直接修改**，所有變更須透過雲端 API。

| 欄位名稱 | 說明 |
| -------- | ---- |
| `instanceId` | 機台實例 ID（由集中式產生，綁定 MAC） |
| `mac` | 網卡 MAC 位址（註冊時綁定） |
| `registeredAt` | 首次向集中式註冊時間 |
| `lastSeen` | 最後上線時間 |
| `status` | 註冊狀態（registered / offline / suspended） |

> **說明**：
> - `instanceId` 由集中式產生，確保唯一性
> - 註冊時 MAC 與 `instanceId` 綁定
> - `machines` 表即為單機本地後台的實例表，用於追蹤資料來源與遊戲連線狀態

## 4. 資料表設計

### 4.1 機台表（`machines`） [L05, C101, L06, C14, L41] (C101: 編輯/刪除)

> **說明**：機台表即為單機本地後台的實例表，必須向集中式註冊才能啟用。

| 欄位名稱       | 資料類型     | 約束條件      | 說明              |
| -------------- | ------------ | ------------- | ----------------- |
| `instanceId`  | VARCHAR(64)  | PRIMARY KEY   | 機台實例 ID（由集中式產生，綁定 MAC） |
| `name`         | VARCHAR(100) | NOT NULL      | 機台名稱          |
| `ip`           | VARCHAR(15)  | UNIQUE        | 機台 IP 位址      |
| `mac`          | VARCHAR(17)  | UNIQUE NOT NULL | 機台 MAC 位址（註冊時綁定） |
| `balance`      | DECIMAL(15,2)| DEFAULT 0.00  | 當前可用餘額      |
| `tags`         | JSONB        | DEFAULT '[]'  | 機台標籤（如：門市、區域、機型等，支援多標籤篩選）|
| `registeredAt` | DATETIME    | NOT NULL      | 首次向集中式註冊時間 |
| `lastSeen`     | DATETIME    |               | 最後上線時間      |
| `status`       | ENUM(...)    | NOT NULL      | 機台狀態（registered/offline/suspended） |
| `hostName`     | VARCHAR(100) |               | 主機名稱          |
| `version`      | VARCHAR(20)  |               | 軟體版本          |
| `lastSyncTime` | DATETIME     |               | 最後同步時間      |
| `createdAt`    | DATETIME     | DEFAULT NOW() | 建立時間          |
| `updatedAt`    | DATETIME     | DEFAULT NOW() | 更新時間          |

#### 標籤（Tags）設計說明

**用途**：為機台新增自定義標籤，支援多維度篩選與分類管理。

**標籤格式**：
```json
{
  "tags": ["台北門市", "A區", "VIP專用", "老虎機"]
}
```

**使用場景**：
- 依門市/分店篩選機台
- 依區域篩選機台
- 依機型分類
- 依客群分類（VIP、一般）
- 自定義分類

**查詢範例**：
- 查詢「台北門市」的所有機台
- 查詢「A區」且「VIP專用」的機台
- 營收報表依標籤維度分析

#### 業務規則

所有資料表都有主鍵索引，確保資料的唯一性與快速查詢。

### 6.2 外鍵索引

所有外鍵欄位都建立索引，加速關聯查詢。

### 6.3 常用查詢索引

- **機台表**：IP、MAC、狀態。
- **遊戲商表**：狀態、分類。
- **遊戲表**：供應商、狀態、類型。
- **機台表**：IP、MAC、狀態、餘額。
- **交易表**：機台 ID、遊戲 ID、交易類型、交易時間。
- **多人遊戲牌桌表**：遊戲 ID、狀態、類型。
- **多人遊戲牌局表**：牌桌 ID、開始時間、狀態。
- **多人遊戲下注表**：牌局 ID、機台 ID、交易 ID。
- **荷官表**：狀態、當前牌桌。
- **荷官操作記錄表**：荷官 ID、牌桌 ID、牌局 ID、時間。

### 6.4 複合索引

- **交易表**：`(machineId, createdAt)`、`(gameId, createdAt)`。
- **機台表**：`(status, createdAt)`。

## 6. 備份與恢復策略

### 7.1 集中式後台備份

每日進行完整備份，備份儲存在獨立的備份伺服器。每小時進行增量備份，支援時間點恢復。備份檔案加密儲存，定期驗證備份完整性。

### 7.2 單機本地後台備份

> **簡化設計**：本地端資料主要為雲端快取，備份需求簡化。

- **本地日誌備份**：本地交易佇列、日誌等資料手動/自動備份
- **雲端同步**：所有重要資料由雲端統一管理，本地僅作暫存
- **加密儲存**：本地備份檔案加密儲存

### 7.3 恢復流程

- **集中式後台**：由資料庫管理員進行恢復，需要驗證備份完整性，恢復前進行備份。
- **單機本地後台**：本地快取可直接從雲端重新同步，無需複雜的本地恢復流程。

## 7. 資料安全

### 8.1 加密

敏感欄位（密碼、交易簽名）使用 **AES-256-GCM** 加密演算法進行加密。資料庫連線使用 SSL/TLS 加密協議。

### 8.2 存取控制

實施嚴格的資料庫使用者權限管理，不同使用者擁有不同的操作權限。應用程式使用專用的資料庫帳號進行連線。

### 8.3 稽核日誌

記錄所有重要操作的詳細日誌，包括操作者、操作時間、操作內容、操作結果等資訊。定期審查稽核日誌，發現異常操作。

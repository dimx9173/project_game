## 1. 文件概述

本文件詳細闡述遊戲平台管理系統的整體系統架構，包含集中式後台與單機本地後台兩個層次的架構設計、資料流、通訊機制等內容。

## 2. 系統架構概述

### 2.1 中央集權架構

遊戲平台採用**中央集權架構**設計，所有單機本地後台必須先向集中式註冊才能啟用。

- **集中式後台**：負責全局資料監控、全平台交易統計、系統設定管理、**機台與玩家註冊認證**。
- **單機本地後台**：受集中式管理的單機系統，負責本地機台、本地遊戲、本地交易結算。

> **核心設計原則**：
> - **單機本地後台**：必須連線才能玩（離線無法開始新遊戲）
> - **玩家**：可跨系統，但註冊時需要連線（離線無法註冊新玩家）

### 2.1.1 運作模式

| 模式 | 說明 | 遊戲狀態 |
| ---- | ---- | -------- |
| **已註冊連線** | 正常向集中式註冊，網路暢通 | ✅ 可開始遊戲 |
| **已註冊離線** | 曾註冊但網路中斷 | ❌ 無法開始新遊戲 |
| **未註冊** | 未向集中式註冊 | ❌ 無法啟用 |

### 2.1.2 玩家註冊

| 項目 | 說明 |
| ---- | ---- |
| **跨系統追蹤** | 玩家可跨系統（不綁定 instanceId） |
| **註冊連線** | 玩家註冊需連線至集中式 |
| **ID 生成** | 集中式統一產生（綁定帳號） |

### 2.2 系統邊界定義

#### 集中式後台邊界

| 邊界元素             | 說明                                     |
| -------------------- | ---------------------------------------- |
| **管理範圍**   | 所有單機本地後台實例、全平台遊戲、全平台交易 |
| **資料角色**   | 全域資料彙總、**機台認證權威**           |
| **可用性要求** | 99.5% 正常運行時間                       |
| **網路依賴**   | 持續連線（核心依賴）                     |
| **職責**       | 全域監控、帳務最終核對、**機台註冊認證** |

#### 單機本地後台邊界

| 邊界元素             | 說明                                   |
| -------------------- | -------------------------------------- |
| **管理範圍**   | 單一實例（單一機台）、本機遊戲、本地交易 |
| **資料角色**   | 本地緩衝（需連線集中式才能運作）       |
| **可用性要求** | **必須連線**（離線無法開始遊戲）       |
| **網路依賴**   | 持續連線（離線則停用）                 |
| **職責**       | 本地遊戲執行、本地交易紀錄             |

> **重要**：單機本地後台為**受集中式管理的客戶端**，**離線後無法開始新遊戲**。

#### 邊界協定

| 協定類型               | 內容                                |
| ---------------------- | ----------------------------------- |
| **通訊協定**     | HTTPS + WebSocket                   |
| **資料交換格式** | JSON                                |
| **認證方式**     | JWT Token（集中）、PIN Code（本地） |
| **連線要求**     | **持續連線**（離線則停用）          |
| **註冊機制**     | 首次啟用需向集中式註冊取得授權       |

## 3. 集中式後台架構

### 3.1 前端架構

集中式後台前端採用 **Blazor Server + Radzen Blazor + Bootstrap 5**。使用 Blazor Router 進行路由管理，Radzen Blazor 提供豐富的 UI 元件，Chart.js 用於資料視覺化，Blazor State 進行狀態管理。支援響應式設計，適配桌面、平板、手機等多種裝置。

### 3.2 後端架構

集中式後台後端採用 **ASP.NET Core 8** 框架。提供 RESTful API 介面，支援 SignalR 實時通訊。使用 PostgreSQL 資料庫進行資料持久化。

### 3.3 核心模組

集中式後台包含以下最高規的核心模組：

1. **儀表板模組**：全平台的自訂 Widget 戰情室。
2. **機台心跳深度監測模組**：(旗艦功能) 監控 CPU/RAM 與網路延遲，提早預防當機。
3. **遊戲與版本管理模組**：涵蓋遊戲授權與跨站點維護。
4. **OTA 版本發布中心**：(旗艦功能) 全球遠端熱更新腳本上傳與無縫派發。
5. **玩家管理模組**：VIP 標籤與中央級距錢包。
6. **錢包交易模組**：全域金流流水帳與圖表化分析。
7. **爭議對帳處理專區**：(旗艦功能) 處理斷線異常、人工介入核銷專區與會計除錯。
8. **串接遊戲管理模組**：高度解耦的 PaymentRouter 設定與多家廠商擴展平台。
9. **多人遊戲模組**：多人遊戲（百家樂、龍虎、骰寶）的集中式管理，牌桌狀態監控與牌局歷史查詢。
10. **使用者權限模組**：完全動態自定義的 RBAC 角色系統 (如：審計員、純查帳員)。
11. **介面設定模組**：玩家端與客製化版面的全域配置管理。

## 4. 單機本地後台架構

> **設計原則**：單機本地後台為雲端快取端點（唯讀），不直接修改資料，所有操作透過雲端 API 執行。

### 4.1 前端架構

單機本地後台前端採用與集中式後台相同的（**Blazor Server + Radzen Blazor + Bootstrap 5**），確保設計風格與使用體驗的一致性。支援本地化部署，可在機台上直接運行。

### 4.2 後端架構

單機本地後台後端採用 **ASP.NET Core 8** 框架。提供 RESTful API 介面，支援 SignalR 實時通訊。使用 SQLite 作為本地資料儲存，僅用於快取與本地日誌。

### 4.3 核心模組

單機本地後台包含以下核心模組：

1. **操作與儀表板模組**：結合開洗分與現場活動捷徑的快搜面板（資料來自雲端快取）。
2. **熱更新接收器模組**：(旗艦功能) 於背景默默接收中央 OTA 封包、MD5 驗證並自我重啟 (不干擾客人)。
3. **深層機台監控模組**：(旗艦功能) CPU、記憶體佔用實時圖表並回傳。
4. **遊戲配置模組**：顯示本地快取的遊戲列表（唯讀，啟用/停用由雲端控制）。
5. **交易管理與非同步佇列模組**：本地交易暫存佇列，連線時同步至雲端（本地不直接寫入）。
6. **系統設定模組**：本地基本設定（網路參數等），重要設定由雲端統一管理。
7. **同步與對帳模組**：檢視雲端同步狀態，無法手動修改雲端資料。
8. **自動對帳系統**：(旗艦強化) 嚴謹地分析本地單據與中央回傳的 Ack 是否完全一致。
9. **機台底層通訊設定**：網路與中心通訊參數、金鑰加密防綁架設定。

### 4.4 本地端限制

> **中央集權架構**：本地端無法直接修改資料，確保資料一致性。

| 功能 | 本地端權限 | 說明 |
| ---- | :--------: | ---- |
| 遊戲管理 | ❌ 唯讀 | 遊戲列表由雲端統一管理 |
| 玩家管理 | ❌ 唯讀 | 玩家資料由雲端統一管理 |
| 系統設定 | ⚠️ 部分 | 僅本地網路參數，重要設定由雲端控制 |
| 交易紀錄 | ⚠️ 佇列 | 本地暫存，連線時上傳雲端 |
| 機台註冊 | ❌ | 必須連線至雲端註冊才能啟用 |
| 遊戲開始 | ❌ | 必須連線至雲端才能開始新遊戲 |

## 5. 資料同步與對帳機制

### 5.1 連線驗證機制

> **核心設計**：單機本地後台**必須連線至集中式才能開始遊戲**。
> **本地端性質**：本地端為雲端快取端點，資料由雲端統一管理，本地無法直接修改。

| 狀態 | 遊戲 | 交易 | 本地資料 |
| ---- | ---- | ---- | -------- |
| **已註冊連線** | ✅ 可開始 | ✅ 可進行 | 雲端同步 |
| **已註冊離線** | ❌ 禁止開始 | ⚠️ 僅顯示歷史 | 唯讀快取 |
| **未註冊** | ❌ 禁止啟用 | ❌ 無法使用 | 無法使用 |

### 5.2 同步流程

```
單機本地後台連線 → 驗證註冊狀態 → 上傳本地交易佇列 → 下載集中式快取更新 → 完成
```

> **同步原則**：本地端**僅上傳交易資料**，下載雲端快取（遊戲、玩家、設定等），不直接寫入資料表。

### 5.3 離線處理

當單機本地後台離線時：

- **禁止開始新遊戲**：離線期間無法開始新遊戲（保護帳務）
- **保留歷史資料**：本地緩存的交易紀錄可檢視
- **連線恢復**：重新連線後同步至集中式

⚠️ **核心原則**：離線期間**禁止開始新遊戲**，確保帳務一致性。

### 5.4 對帳機制

系統定期進行對帳檢查，驗證本地資料與集中式資料的一致性。對於不一致的項目，記錄並標記為待審核，由管理員手動處理。

### 5.5 遠端控制機制

集中式後台可對單機本地後台進行遠端控制（需本地在線）。

**操作流程**：

```
1. 選定單機本地後台 → 選擇目標機台
      ↓
2. 驗證在線狀態 → 確認可連線
      ↓
3. 執行遠端操作 → 開分/洗分/重啟/設定等
      ↓
4. 套用並回傳結果 → 本地執行並回報
      ↓
5. 記錄日誌 → 儲存至集中DB
```

**適用範圍**：

| 功能模組 | 是否支援遠端控制 |
| -------- | :--------------: |
| 機台管理（開分/洗分/重啟/配置） | ✅ |
| 遊戲管理（列表/配置/測試） | ✅ |
| 玩家管理（狀態/錢包） | ✅ |
| 系統設定 | ✅ |

## 6. 安全性架構

### 6.1 認證與授權

系統採用帳號密碼認證，使用 JWT Token 進行會話管理。全面實施基於角色的動態存取控制（Dynamic RBAC），讓集團業主建立完全自定義、顆粒度細緻的操作權限。

### 6.2 資料加密

所有敏感資料（玩家帳號、交易紀錄、配置資訊）使用 **AES-256-GCM** 加密演算法進行加密儲存。通訊採用 HTTPS 加密協議，確保資料傳輸安全。

### 6.3 交易簽名驗證

所有交易使用 **RSA** 數位簽名進行驗證，確保交易的真實性與完整性，防止交易被竄改。

### 6.4 稽核日誌

記錄所有重要操作的詳細日誌，包括操作者、操作時間、操作內容、操作結果等資訊，支援追蹤與審計。

## 8. 部署架構

### 8.1 集中式後台部署

集中式後台部署在雲端或資料中心。前端與後端服務皆使用容器化部署（Docker），支援自動擴展。資料庫使用主從複製實現高可用。

### 8.2 單機本地後台部署

單機本地後台與遊戲引擎部署在同一台機器上。支援 Windows、Linux、macOS 等作業系統。使用 Node.js 執行環境，SQLite 作為本地資料庫。

### 8.3 網路拓撲

- 集中式後台與單機本地後台通過 Internet 連線。
- 單機本地後台與遊戲引擎通過本地網路連線。
- 玩家端應用程式與遊戲引擎通過本地網路連線。

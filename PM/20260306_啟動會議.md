## 🎤 PM 講稿與引言 (Script)

### 1. 開場與專案背景 (00:00)
「大家午安，我是這個專案的 PM。今天把大家找來，是為了我們即將全面啟動的『遊戲平台管理系統 V8 旗艦版』。
相信大家禮拜四都已經收到並大致翻閱過我寄出的 SA/SD 與規格書了。
這次專案對我們影響非常大，我們等於是徹底揚棄了過去的小型網頁後台思維，將整個系統架構遷移到了 **C# .NET 8 的企業級生態**。
我們的目標很明確：**打造一個能夠承受高併發交易、具備自動化遠端派發 (OTA) 能力，且在網路斷線等極端情形下『帳務絕對不能錯』的雙層管理系統**。」

### 2. 技術與核心規格導讀 (00:10)
「這次的 SA 文件裡面有極大的變動。
首先，我們的後端與前端將使用 **ASP.NET Core 8 Web API 搭配 Blazor Server**，資料庫也轉向支援更穩定的 **PostgreSQL (中央) 與 SQLite (單機)**。
在業務規格的部分，我想請大家特別關注我們新增的三個『旗艦模組』。
第一是 **OTA 遠端發布中心**。以後我們的機台不用派人帶隨身碟去修，可以靜默下載並比對 MD5 後自己熱更新。
第二是 **爭議對帳處理專區**。這是會計部門強力要求的，以後只要斷線造成中央與單機兩邊報表對不上，系統必須自動抓出來，並且由人工去下達沖銷指令。這點在開發金流與重試隊列 (Exponential Backoff) 時，請後端團隊務必謹慎對待。
最後是 **PaymentRouter** 的解耦，未來我們預計要接數十家的國外遊戲廠商，所以這套標準化介面必須寫得夠穩。」

### 3. 時程與團隊協作期望 (00:40)
「在後續的時程安排上，我會先請大家這幾天用心地檢視 SA 和 SD 裡的 `Business Rules`（業務規則）與資料表設計是否有矛盾之處。
如果有覺得不合理、或者技術上實作會有巨大風險的地方，請務必在下週一的 Stand-up 中提出來。開發前改規格，永遠比寫出雷再修來得便宜。」

### 4. 總結交棒 (00:50)
「以上是我針對專案目標的引言。接下來，針對這幾份 SA/SD 與規格書，有沒有哪位工程師看了之後覺得：『這個架構圖哪裡怪怪的』或是『這個 API 反應時間要求可能達不到』的？都可以現在提出來。」
